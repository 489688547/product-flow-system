# 销售异常自动修复 API v1

## 目的与边界

`POST /api/platform/v1/data-services/sales-repair` 只处理最新销售事实日疑似截断的自动补拉。销售口径固定为订单创建时间、`Asia/Shanghai`，日完整性规则固定为最新日 GMV 与销量同时低于此前最多 7 个有效日中位数的 25%，且至少有 3 个基准日。

该接口不是历史任意日期重写工具，也不替代快麦官方文件历史补数。快麦订单 API 不含完整退款明细；已有退款事实的日期只能重新导入官方文件。

## 认证与授权

- 必须具有有效公司会话。
- 仅总经办或运营部的非只读身份可以触发。
- 服务端重新读取 D1 并复核异常，不能信任客户端判定或操作者字段。
- Provider 凭据只通过共享平台保险箱解析，浏览器、响应、任务记录和日志均不得出现明文。

## 请求与响应

请求只接受一个自然日：

```json
{ "date": "2026-07-21" }
```

首次建立任务返回 HTTP 202：

```json
{
  "synced": true,
  "scheduled": true,
  "run": {
    "id": "kuaimai-sales-repair:2026-07-21:sales-completeness-v1",
    "date": "2026-07-21",
    "status": "running",
    "attempts": 1
  }
}
```

同日运行中、成功、需要人工处理或已失败两次时返回 HTTP 200 和现有安全任务摘要，不建立新任务。服务端判定当天已不异常时返回 `scheduled: false, reason: "not_anomalous"`。

## 执行、幂等与写入

- 幂等键由服务端固定为 `kuaimai + date + sales-completeness-v1`，客户端不能覆盖。
- 同一天自动尝试上限为 2；运行中、成功和 `manual_required` 不重复调度。
- 后台任务通过 Pages `waitUntil` 执行，从快麦第 1 页按创建时间拉取，最多 40 页；只有 `hasNext=false` 才视为完整。
- 拉取结果先在内存暂存。存在退款字段、分页不完整、结果仍低于基准、行超过 2000 或字段无效时，不删除也不覆盖旧事实。
- 安全通过后，删除异常日并插入完整日聚合和元数据必须放在同一个 D1 batch 中。
- `data_sync_runs` 使用受保护的 `systemSyncRuns` 实体类型追加安全任务摘要，避免普通数据中心整包保存删除系统任务。

## 状态与可观测性

任务状态为 `running`、`success`、`failed` 或 `manual_required`。记录日期、尝试次数、开始/完成时间、行数、修复前后 GMV/销量比例、稳定错误码和安全中文说明；不记录订单明细、凭据、Provider 原始响应或 SQL。

稳定错误见 `docs/platform/error-codes.md`。所有失败都保留旧销售事实；调用方刷新数据中心状态即可读取任务，不需要额外轮询接口。

## 兼容、容量与回滚

- 不新增本接口专属变量、Secret、绑定或 D1 表；控制任务复用正式控制库，销售事实与业务同步摘要使用服务端选定的业务库。展示环境写入复用统一销售两倍转换，客户端不能选择绑定或数据库 ID。
- 旧客户端继续只读销售数据，不会自动触发修复。
- 回滚路由和前端后，已完成的销售事实与追加任务记录继续有效；失败任务不需要数据恢复。
- 生产验证必须分别确认公司会话权限、快麦真实调用、D1 写入、数据总览刷新和同步记录，不得用本地构建代替。
