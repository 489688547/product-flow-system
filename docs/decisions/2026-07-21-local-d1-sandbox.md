# 本地开发增加本地 D1 沙箱模式作为远程绑定故障兜底

- 状态：已采纳
- 日期：2026-07-21

## 背景

`npm start` 的本地在线模式依赖 wrangler 远程 D1 绑定。2026-07-21 远程绑定在 dev 代理下持续返回 `D1_ERROR: internal error`（CLI 直查正常），导致本地登录与所有数据查询不可用，开发完全阻塞。远程绑定故障期间，开发者没有任何不连生产库又能验证登录与页面的替代路径。

## 决策

1. 新增沙箱模式：`npm start -- --local-d1`（或 `LOCAL_D1_SANDBOX=1`）使用 `wrangler.local.toml` 指向本地 D1，不触生产数据。
2. Pages dev 不支持自定义 `--config` 路径，启动脚本临时把沙箱配置换入 `wrangler.toml`，退出时（含异常退出后的下次启动）自动恢复线上配置；磁盘上长期保留的必须是线上版本。
3. `scripts/seed-local-sandbox.mjs` 负责本地迁移与播种最小可用身份（active executive + 哈希个人令牌），不写入任何真实凭据。
4. 沙箱模式只用于本地验证登录与页面行为；涉及真实钉钉、快麦等外部平台动作的验证仍走本地在线模式。
5. 线上 Pages 部署配置、绑定与环境变量不因沙箱模式改变。

## 备选方案

- 等待 Cloudflare 修复远程绑定：阻塞时间不可控，拒绝。
- 直接改 `wrangler.toml` 指向本地 D1 并手工还原：容易误提交线上配置，拒绝。
- 在 CI 用本地 D1：已有测试套件覆盖，沙箱解决的是人工登录验证，不重复建设。

## 影响

- 远程 D1 绑定故障时本地开发不再阻塞；沙箱数据只影响本机。
- 启动脚本增加配置交换与恢复职责，恢复失败会在下次启动时重试。
- `wrangler.local.toml` 与播种脚本需随迁移结构演进同步维护。

## 回滚

删除 `wrangler.local.toml`、`scripts/seed-local-sandbox.mjs` 与启动脚本中的 `--local-d1` 分支即可；线上部署路径不引用沙箱配置。
