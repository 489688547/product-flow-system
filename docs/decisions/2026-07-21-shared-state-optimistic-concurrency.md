# 共享整状态采用先读后写、乐观版本与写前快照

- 状态：已采纳
- 日期：2026-07-21

## 背景

产品全周期历史实现把浏览器本地缓存与 D1 公司状态视为可互相覆盖的同级副本。旧标签页发现本地脏标记后会在读取服务器前 POST 整份状态；服务端删除并重建所有分片，不校验客户端看到的版本，也不生成普通写入快照和审计。2026-07-20 至 2026-07-21，该路径多次把默认状态写入生产 D1，覆盖真实产品进度。

## 决策

共享整状态采用服务器权威的先读后写协议：

1. 客户端必须先 GET 并保存服务器 `updatedAt`，没有基线时不能构造写请求。
2. POST 必须提交精确 `baseUpdatedAt`；缺少或落后都在写入前返回 409。
3. 浏览器脏缓存和代码默认状态不再自动上传，只能保存为本机恢复副本。
4. 每次接受写入前保存完整快照并创建审计，写入后记录新版本和结果。
5. 事故恢复继续使用已有生产数据网关，不新增数据库旁路。

## 备选方案

- 只修客户端：无法阻止已发布旧标签页、旧分支或其他调用方，拒绝。
- 只做数量下降检测：合法删除与事故覆盖难以稳定区分，且不能解决并发，拒绝。
- 拆成逐实体事件存储：长期更强，但迁移范围过大，不适合本次事故修复；待出现新的真实扩展需求后另立 ADR。

## 影响

- 旧客户端写入会收到 409，刷新到新版本后恢复。
- 多标签页从同一基线开始时只有先写者成功，后写者必须刷新。
- 普通自动保存会增加快照与审计写入；现有快照上限负责容量控制。
- D1 中其他业务表不受产品状态恢复影响。

## 回滚

代码可回滚到上一 Pages 部署，但不得重新开放无基线写入。若必须回滚客户端，应保留服务端门禁并将旧客户端视为只读。错误的数据恢复通过关联审计的写前快照回滚，不执行整库 Time Travel。
