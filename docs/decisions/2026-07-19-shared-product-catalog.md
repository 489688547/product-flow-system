# ADR：由数据中心维护共享商品主数据

- 日期：2026-07-19
- 状态：已接受

## 背景

产品全周期、供应链和销售曾分别保存产品名称、关系和 69 码。快麦商品档案既可由开放平台读取，也可从商品档案 V2 导出，但两条通道覆盖不同：实测 API 返回 912 条、合并为 855 个主商品；当前商品档案导出有 298 条规格记录、274 个商品，并补充了 API 未返回的规格商家编码、69 码和供应商字段。

如果各 App 分别导入，将产生重复编码、关联漂移和成本权限泄露；如果把所有 ERP 商品自动创建为产品项目，又会混淆“业务产品”和“ERP 商品”。

## 决策

1. 新增平台级商品目录，由数据中心负责快麦同步、文件导入和质量治理。
2. 商品优先用来源 + 主商家编码形成稳定 ID，SKU 优先用商品 ID + 规格商家编码形成稳定 ID；系统商品/SKU ID 作为来源属性保留。
3. 快麦 API 与商品档案文件按非空字段和 SKU 集合幂等合并。API 未返回文件商品时不标记删除；只有未来获得能覆盖同一全集的来源后才启用缺失判定。
4. 产品全周期只保存 `catalogProductId`，继续保留自己的阶段、负责人和历史 `skuCodes`；不会自动为所有 ERP 商品创建产品项目。
5. 供应关系保存 `catalogProductId` 和可选 `catalogSkuId`，保留旧 `productId` 兼容。
6. 目录通过 `/api/platform/v1/product-catalog*` 和独立 D1 表提供。全员可读基础字段，采购成本只返回给总经办、财务、供应链和采购范围。
7. 商品经营销量扩展同一目录读模型：销售事实继续唯一保存于 `product_sales_daily`，服务端按日期、平台和 SKU 编码聚合关联；不建立商品销量复制表，也不把销量写入商品 payload。
8. 快麦订单自动同步以官方 `timeType=created` 和响应 `created` 为下单时间口径。默认经营统计排除“其它/未知”平台，歧义编码不猜测归属。

## 后果

- 产品 App、供应链和供应商管理读取同一套商品和 69 码。
- 文件导入与 API 自动同步不会因来源身份不同形成两套商品。
- 目录增加四张 D1 表、迁移、环境能力和平台契约维护成本。
- 商品目录读取可按最多 370 天扫描并在 D1 按编码/平台聚合销售事实；前端不接收订单明细，容量继续由现有销售日聚合表承担。读取接口不承担 D1 建表或改表。
- 自动同步当前不能单独判断 ERP 删除；状态字段保留在数据层和质量治理中，商品经营主表移除状态列与状态筛选，删除判断延后到有完整同口径来源时。

## 迁移与回滚

- 迁移新增 `product_catalog_items`、`product_catalog_skus`、`product_catalog_sync_runs`、`product_catalog_meta`，不改旧表。
- 首次上线可先导入商品档案文件，再执行快麦 API 同步；顺序相反也按稳定 ID 合并。
- 回滚前端时移除 Provider、数据中心入口和关联 UI，保留目录表；旧 `skuCodes` 和 `productId` 仍可被旧版本读取。
- 不执行反向快麦写入。生产导入或同步失败时保留上次成功目录。
- 回滚销量视图只移除查询参数和页面销量列，保留商品目录与销售日事实；历史销售可继续按原回填链路校准。
