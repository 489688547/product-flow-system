# ADR：数据中心共享加密凭证保险箱

- 状态：接受
- 日期：2026-07-19

## 背景

数据中心需要接入电商、广告、ERP、NAS、邮箱和其他内部系统。部分平台有 API，部分依赖已登录网页或文件导出，部分登录需要密码、Token、Cookie、扫码或验证码。只登记网页地址无法支持无人值守每日采集；把明文账号散落在钉钉表、浏览器、脚本和各业务 App 也无法统一权限和审计。

加密凭证至少有三个真实消费者：经营数据连接器、内部系统保险箱和固定公司 Mac 上的采集器，因此属于共享平台能力，不应由数据中心页面自行保存或解密。

## 决策

1. 在 D1 保存凭证密文及权限、版本和审计元数据，使用 Cloudflare Secret `PLATFORM_CREDENTIAL_MASTER_KEY` 和 AES-256-GCM 加密；旧名 `DATA_CREDENTIAL_MASTER_KEY` 仅作为迁移期兼容别名。
2. OTP、短信验证码、二维码内容、滑块答案和当次人工验证结果永不持久化；密码、API Secret、Token、Cookie 和可复用会话可以加密持久化。
3. 普通列表和详情 API 永不返回明文。替换、查看/复制、采集器取用和权限管理是不同动作权限。
4. 明文查看需要近期认证、条目范围授权、明确用途、禁止缓存和追加式审计。
5. 采集器不持有主密钥；服务器只为被分配任务签发短时、最小字段范围、默认一次性消费的 task grant，并使用采集器公钥封装。
6. 连接器定义和平台专属表单位于数据中心功能边界；加解密、授权和审计位于 `functions/api/platform/_shared`，稳定 API 使用 `/api/platform/v1/credential-vault`。
7. 钉钉 AI 表格“账密表”仅作为一次性迁移来源；先预览分类与冲突，再直接加密写入，不保留明文中间文件或持续同步。

## 采用该方案的原因

- D1 泄漏时攻击者不能仅凭数据库内容得到明文；Cloudflare Secret 与数据库分离。
- 所有业务 App 和采集器复用同一套动作权限、审计和密钥轮换规则。
- 平台专属登录差异保留在连接器 schema 中，不污染共享保险箱 API。
- 人工验证码与可复用凭证分开，既支持实际登录流程，也不尝试绕过平台风控。

## 未采用方案

### 不保存任何凭证，只依赖浏览器长期登录

登录失效后无法可靠恢复，也不能覆盖 API、NAS、邮箱或其他内部系统，因此不能支持统一每日采集。

### 在 D1 保存明文，依赖“仅内部使用”

数据库备份、调试、越权查询或日志错误会直接暴露全部账号，风险不可接受。

### 把主密钥放在采集器或前端

任一设备或浏览器泄漏都会扩大为全库解密能力，且无法实现最小任务范围授权。

### 每个平台单独实现加密与审计

会产生不同算法、权限和日志行为，密钥轮换及安全修复难以一致完成。

## 后果

- 需要新增 D1 迁移、Cloudflare Secret、环境能力登记、平台 API 契约、权限和审计测试。
- 生产部署必须在迁移和 Secret 就绪后才开放凭证写入。
- 密钥丢失会导致密文无法恢复，因此需要受控备份和版本化轮换流程。
- 服务器在 reveal/task grant 的短时间内会接触明文，必须禁止请求体日志、限制响应缓存并缩小代码边界。
- 内置平台卡片只能表示可配置；真实接入状态仍取决于适配器、采集器和同步证据。

## 回滚

停用连接器、reveal 和 task grant 路由，吊销机器令牌并保留加密记录与审计。回滚不得删除密文或覆盖主密钥；恢复旧版本前必须确保它仍识别当前 `key_version`。如需彻底退出功能，先导出受控加密备份，再按公司数据销毁流程处理 D1 记录。
