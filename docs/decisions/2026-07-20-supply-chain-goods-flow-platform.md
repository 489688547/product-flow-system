# 架构决策：供应链管理作为唯一入口，货流抽取共享数据平台

## 状态

已接受，2026-07-20。

## 背景

现有供应链管理已经拥有供应商、采购付款、产品供应关系、库存盘点和质量问题。新增货流能力需要进一步管理需求波动、生产在途、库存效率、履约、逆向质量和现金循环。如果直接新增一个“货流 App”，同一用户会面对两套入口，同一商品、供应商、库存和付款也容易形成重复状态。

另一方面，把 CCC、每日库存、月度盘点校准和跨来源事件继续塞进供应链整包状态，会阻止产品全周期、数据中心、经营驾驶舱和公司 AI 复用同一事实，并扩大并发覆盖、迁移和审计风险。

## 决策

1. “供应链管理”保留为唯一业务 App 入口，页面内部升级为货流驾驶舱和七环节左侧 Tab。
2. 现有供应商、采购付款、产品供应链、库存盘点和质量数据保留并迁移或兼容读取，不复制一套货流档案。
3. 抽取独立货流数据平台，拥有跨来源不可变事件、库存日投影、盘点确认、平台账期、CCC 版本、异常和审计。
4. 新稳定契约使用 `/api/platform/v1/goods-flow/*`；供应链、产品全周期、数据中心、经营驾驶舱和公司 AI 按权限读取同一数据。
5. 业务工作流和人的判断归供应链 App；外部适配、统一主键、确定性计算、权限、版本和审计归货流平台。
6. 现有 `/api/supply-chain` 在迁移期保持兼容，通过功能开关逐步切换读模型，不做一次性破坏性替换。

## 备选方案

### 继续扩展现有供应链整包状态

首期改动较少，但跨 App 复用、可追溯事件、历史重算和并发写入都会继续受限，因此拒绝。

### 新建独立货流 App 和独立数据

概念边界清晰，但会产生第二套入口、商品、库存和供应商事实，用户无法判断该去哪个页面，因此拒绝。

### 一个业务入口与独立数据平台

保留现有使用习惯，同时把跨来源事实和指标抽取为稳定能力，采用本方案。

## 后果

### 正面

- 供应链人员不需要在两个 App 之间切换。
- 产品、数据中心和 AI 可以读取统一 SKU、69 码、库存与 CCC。
- ERP、钉钉、快麦和盘点文件的差异能够在来源层审计。
- 月度指标可冻结、重算和比较，不覆盖历史结果。

### 成本

- 需要维护兼容投影、迁移对账和功能开关。
- 新平台 API 必须建立权限、错误码、可观测性和契约测试。
- D1 新表、生产数据访问和外部 provider 适配必须同步更新环境能力与集成注册表。

## 兼容、迁移与回滚

- 新表和 API 以旁路方式增加，不删除旧供应链状态。
- 首次回填只生成对账结果；无法确定性映射的记录进入异常队列。
- 新页面按功能开关读取货流平台；关闭开关恢复旧读模型。
- 回滚停止新事件和投影写入，保留历史表用于审计；不得删除已确认盘点、付款或冻结指标。
- 快麦库存接口未验证前，库存继续标记为 ERP/文件快照，不登记为已连接自动能力。
