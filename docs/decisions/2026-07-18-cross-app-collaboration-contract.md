# ADR：跨 App 部门协同使用共享契约与平台投影

## 状态

已接受，2026-07-18。

## 背景

产品全周期已经向经营执行平台上报进度、里程碑、风险和负责人变化；供应链、数据中心和品牌内容又分别出现异常库存、质量整改、数据质量、内容交接和投放判断。若每个 App 自己实现跨部门接收、退回、阻塞、验收、权限和钉钉同步，公司会得到多套责任口径和重复待办。

当前 `/api/platform` 只面向总经办整包保存战略执行状态，不适合普通部门按参与范围读取和修改协同事项。业务 App 又不能直接调用外部平台或把自身业务状态交给平台控制。

## 决策

抽取共享的部门协同能力：

1. 业务 App 保持业务事实所有权，并通过局部纯函数适配器生成标准 `CollaborationDraft`。
2. 平台拥有协同事项、唯一责任、状态机、权限、追加活动、老板行动投影和钉钉同步。
3. 新增 `/api/platform/v1/collaboration-items` 细粒度契约和独立 D1 表；普通员工只读取本人或本人部门参与范围。
4. 老板行动台通过协同事项和既有战略/项目事实读取时投影，不建立第二份可编辑老板事项表。
5. App 适配器不持久化、不流转、不发送外部动作；异常判断继续留在各 App 领域模块。
6. 现有 `appEvents` 保持兼容，不自动批量转为协同事项。

## 备选方案

### 每个 App 独立建设协同

首个 App 开发较快，但会重复状态、权限、审计、待办和老板汇总，并导致同一部门在多个页面使用不同责任规则，因此拒绝。

### 所有跨部门任务集中到一个通用任务系统

责任统一，但容易复制业务字段、失去来源上下文，并让平台反向控制业务状态，因此拒绝。

### 继续使用战略整包状态

改动较少，但无法安全支持普通部门，也会重现整包容量、并发覆盖和权限过宽问题，因此拒绝。

## 后果

### 正面

- 四个真实 App 共用一致的责任、状态、权限、审计和钉钉规则。
- 普通员工不需要访问总经办战略整包状态。
- 老板只看符合升级规则的异常和决策，不接收全部日常任务。
- 每个 App 仍可独立发布、回滚和演进业务字段。

### 成本

- 需要维护稳定 v1 契约、参与者索引、乐观锁和 App 适配器。
- 业务事实变化与协同证据快照可能不同步，页面必须展示事实时间和返回来源入口。
- 首版依赖用户确认创建，服务器自动摄取需要后续单独设计。

## 兼容与回滚

- 新表和新 API 不修改现有 `/api/platform`、`/api/state` 和 App 业务表。
- 功能开关关闭后恢复原公司首页和导航，停止新建与钉钉同步；历史协同记录保留。
- 破坏性契约变化必须通过新 API 版本提供。
