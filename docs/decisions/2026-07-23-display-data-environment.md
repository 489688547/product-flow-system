# 对外展示采用独立单展示数据库与统一数据环境路由

- 日期：2026-07-23
- 状态：已采纳
- 关联功能：`docs/features/demo-data-environment/`

## 背景

系统需要对外展示真实的业务结构和操作关系，同时把销售规模调整为正式数据的两倍。页面临时乘二会让总览、详情、导入、同步和 AI 互相矛盾；把展示数据写回正式库会污染经营事实。当前只有一名开发和一个最高权限账号，但这不降低数据、凭证和外部平台动作的隔离要求。

## 决策

1. 保留 `PRODUCT_FLOW_DB` 作为正式控制库和正式业务库，新增独立物理 D1 `DEMO_FLOW_DB` 保存展示业务数据。
2. 服务端认证后统一解析当前浏览器会话的数据环境，把 `controlDb`、`businessDb` 和环境版本注入业务路由；浏览器不能选择 binding 或数据库 ID。
3. 采用一个展示库，不采用 A/B 双槽。刷新期间展示环境整体维护，失败后保持不可用，不回退正式业务库。
4. 展示刷新只复制目录白名单。个人字段确定性脱敏，销售可加总事实乘二，比例和均值重算；未知表、凭证、会话、令牌和秘密审计默认跳过。
5. 采集控制任务写正式控制库并固化目标环境与版本，真实读取结果只投影到选定业务库。
6. 展示环境中的外部写动作必须经过登记的模拟器，保留校验、权限、幂等和安全审计，但不加载真实写凭据、不调用 Provider。
7. AI Provider 配置、租约、Token 和 Skill 次数始终写控制库；业务 Context 和 Skills 使用选定业务库，审计独立记录 `data_environment`。
8. 展示数据库用于对外展示和真实能力验证，不是测试环境。试验性写入继续使用本地 SQLite 沙箱。

## 未采用方案

- 单库环境字段：需要改写几乎所有表、索引和查询，遗漏环境条件会直接串库。
- 查询时乘二：无法维持编辑、导入、同步、历史记录、跨 App 关系和 AI 的一致事实。
- A/B 双槽：能够无停机刷新，但增加一倍存储、绑定、切换和回收复杂度；当前用户接受刷新时等待。
- 为每个分支或员工建数据库：增加配置漂移和运维成本，不能解决统一规则写回。

## 后果

- 业务存储必须依赖中间件选择的 `businessDb`，直接使用 `PRODUCT_FLOW_DB` 只允许在控制面清单内。
- 新表必须在展示目录声明复制策略；新采集任务必须保存目标版本；新外部写必须声明展示行为；新 AI 业务 Skill 必须读取 `businessDb`。
- 刷新是分步、游标化、租约保护且可续跑的原地重建。用户在刷新期间不能浏览展示环境。
- 正式和展示 D1 必须执行兼容迁移，并由环境能力检查验证两套 binding。
- 正常业务页不显示展示标记，因此切换、更新时间、维护和失败信息必须在设置页完整表达。

## 回滚与恢复

应用级回滚先在正式控制库关闭展示环境并撤销环境授权，使所有请求回到正式业务库；不得把失败的展示请求静默改读正式库。代码回滚移除展示路由、设置入口和 `DEMO_FLOW_DB` 绑定，不修改正式业务事实。展示库损坏时重新运行全量刷新；需要运维级恢复时使用 Cloudflare D1 Time Travel，控制令牌不进入应用配置、日志或仓库。
