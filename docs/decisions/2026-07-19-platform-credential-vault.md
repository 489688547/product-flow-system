# 架构决策：应用内平台凭据保险箱

## 状态

已接受

## 背景

环境就绪检查能够识别钉钉和快麦配置缺失，但业务管理员没有应用内处理入口。全部依赖 Cloudflare 后台不适合老板日常维护；让应用持有 Cloudflare 管理令牌又会扩大基础设施控制权限。系统已经使用 Pages Functions、D1、Web Crypto 和公司会话权限边界。

## 决策

使用一个 Cloudflare Secret `PLATFORM_CREDENTIAL_MASTER_KEY` 作为 AES-256-GCM 主密钥，把每个平台的当前公司连接加密保存在 D1。浏览器只获得字段存在性和验证状态。候选配置通过平台只读测试后才替换当前版本。服务端解析顺序为启用的保险箱连接优先、旧环境变量回退。

## 备选方案

- 只在 Cloudflare 后台维护：安全边界简单，但老板无法在产品内闭环处理阻断，环境漂移继续存在。
- 应用调用 Cloudflare API 修改 Secret：需要长期保存高权限控制面令牌，故障与权限半径过大。
- D1 明文保存：无法满足凭据泄漏和数据库读取风险要求，不采纳。

## 后果

- 收益：老板可以直接处理连接；钉钉、快麦和环境检查使用同一来源；所有更新可审计。
- 成本：部署必须管理一个根密钥，并维护密钥版本和恢复说明。
- 风险：根密钥丢失会使保险箱密文不可恢复；旧环境变量回退和重新录入提供恢复路径。
- 运维：主密钥只配置一次；平台凭据日常更新在数据中心完成。

## 兼容与迁移

不读取或自动迁移旧环境变量。新表为空时行为与当前版本一致。新连接异常时可以停用或回滚代码，旧变量继续生效；D1 新表保留。
