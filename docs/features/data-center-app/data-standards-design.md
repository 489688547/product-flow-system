# 数据中心“数据口径”治理与计算设计

## 背景与目标

数据中心现有“指标管理”页面只有净销售额和毛利两条只读说明，数据总览仍在功能代码中直接计算部分指标。这会造成“页面登记一套、实际算法使用另一套”的风险。

本功能把页面统一更名为“数据口径”，建立可新增、查看、修改、归档、版本化、预览和重算的公司级口径能力。数据中心总览只能展示已发布口径的计算结果；供应链、经营驾驶舱、公司 AI 和其他业务 App 后续也通过稳定 `metricCode` 读取同一套定义与结果。

## 非目标

- 不允许用户输入或执行任意 SQL、JavaScript 或其他代码。
- 不把货流 v1.0 的 Phase 1、Phase 2 指标提前标记为已生效口径。
- 不因口径修改静默覆盖历史结果。
- 不物理删除已发布口径、版本、计算结果或审计记录。
- 不在本次改变钉钉、快麦、ERP 等外部平台的接入方式。

## 平台能力决策

结论：`抽取共享能力`。

口径定义、版本选择、公式校验、计算结果和重算任务属于多个真实 App 共用的稳定契约，不继续隐藏在数据中心 React 页面或单一状态对象中。共享能力使用 `/api/platform/v1/data-standards` 契约；数据中心负责治理与展示，但不拥有一套私有算法。

现有 `metricDefinitions` 作为兼容输入迁移到共享口径服务。迁移完成前，旧 `/api/data-center` 仍可返回兼容字段；新写入只进入共享服务，避免双写分叉。

## 首批口径目录

### 销售口径

| 指标 | metricCode | 定义与公式 | 统计规则 | 责任部门 |
| --- | --- | --- | --- | --- |
| 净销售额 | `sales.net_sales` | 净销售额按订单创建日汇总 | `create_time`、`Asia/Shanghai`、排除“其它/未知” | 财务部 |
| 毛利 | `sales.gross_profit` | 毛利按订单创建日汇总 | `create_time`、`Asia/Shanghai`、排除“其它/未知” | 财务部 |
| 销量 | `sales.quantity` | 有效订单商品数量求和 | `create_time`、`Asia/Shanghai`、排除“其它/未知” | 运营部 |
| 退款率 | `sales.refund_rate` | `退款金额 ÷ 销售额 × 100%` | 分母为零时结果为空，不补零 | 财务部 |
| 毛利率 | `sales.gross_margin_rate` | `毛利 ÷ 净销售额 × 100%` | 分母为零时结果为空，不补零 | 财务部 |

### 货流口径

| 指标 | metricCode | 定义与公式 | 周期与来源 | 责任部门 |
| --- | --- | --- | --- | --- |
| 库存资金 | `goods_flow.inventory_cash` | `已分摊采购实付 − 累计销售成本 ± 已确认盘点损益` | 日；采购付款、销售成本、已确认盘点 | 财务部 |
| 库存周转天数 | `goods_flow.inventory_turnover_days` | `月均校准库存成本 ÷ 当月销售成本 × 当月天数` | 月；校准库存日快照、销售成本 | 供应链部 |
| 应收天数 | `goods_flow.receivable_days` | 按各平台净销售额对财务配置的有效账期加权 | 月；平台净销售额、平台账期版本 | 财务部 |
| 应付天数 | `goods_flow.payable_days` | 按采购金额加权计算“验收入库日至实际付款日” | 月；采购、入库、付款；缺少入库日时使用审批通过日并标记估算 | 财务部 |
| 现金循环周期 CCC | `goods_flow.ccc_days` | `库存周转天数 + 应收天数 − 应付天数` | 月；引用三个已版本化分量 | 财务部 |
| 断货率 | `goods_flow.stockout_rate` | `核心 SKU 断货天数 ÷ 核心 SKU 应售天数` | 月；核心 SKU 清单、每日可售库存 | 供应链部 |

所有首批口径从 `v1` 开始。未具备所需事实数据的货流指标可以发布定义，但计算状态必须为“数据未覆盖”，不能产生模拟值。

## 口径生命周期

### 新增

有编辑权限的用户选择业务分类、责任部门、名称、稳定 `metricCode`、单位、统计周期、维度、数据来源和公式。`metricCode` 发布后不可修改；需要新的业务含义时必须新建口径。

### 修改与发布

编辑已发布口径会创建新版本，不覆盖旧版本。`metricCode` 永久不可修改；名称、分类、责任部门、单位和周期与公式一起保存为新版本快照。保存即直接发布，不设置额外审核流；每个版本必须指定生效日期。历史周期解析当时有效的版本，新版本不会自动改写历史结果。

同一口径的生效日期必须严格递增，不能有两个版本在同一天生效。并发写入使用乐观版本；版本落后返回 409，客户端刷新后由用户重新确认。

### 归档

“删除”统一实现为停用/归档。归档后不能用于新的周期计算，也不能被新公式引用；历史版本、结果、依赖关系和审计仍可查看。口径存在下游依赖时，页面必须先列出受影响的总览指标和订阅 App。

### 历史重算

用户可以选择口径、开始日期、结束日期和目标版本发起重算。系统先显示预计影响周期、依赖指标、数据覆盖和现有结果数量，用户确认后建立异步任务。

重算生成带新计算批次 ID 的结果，不原地覆盖旧结果。读取接口默认返回当前有效批次，同时保留旧批次用于审计和回滚。失败任务不替换当前有效结果。

## 安全公式模型

公式使用受控抽象语法树保存，由可视化构建器生成。允许的节点包括：

- 已登记事实字段与已发布指标引用；
- `sum`、`average`、`weighted_average`、`count`、`count_distinct`；
- 加、减、乘、除；
- 日期差、有效天数、按日/月周期聚合；
- 等于、包含、范围、空值判断等白名单过滤；
- `coalesce` 等明确的缺失值回退，并附带估算标记。

禁止任意 SQL、脚本、函数名、跨权限字段和未登记数据源。发布前必须完成：schema 校验、字段权限校验、单位检查、循环依赖检查、最大依赖深度检查、除零检查、日期范围限制和样本预览。

公式依赖形成有向无环图。被引用口径归档或变更时，不自动重写上游公式；系统提示依赖并要求用户显式发布上游新版本。

## 数据可信规则

- 缺失数据不补零；无法计算时返回空值和明确原因。
- 每个结果同时保存并返回数据覆盖率、可信等级、估算标识、数据截止时间和口径版本。
- CCC 任一分量缺失时状态为“不完整”，不能按零参与计算。
- 应付天数使用审批通过日替代入库日时必须标记“估算”。
- 盘点只接受已确认盘点损益，不覆盖原始 ERP 快照。
- 预览与正式计算使用同一个编译器和执行器，避免预览结果与正式结果不一致。

## 权限

- 总经办非只读身份可以管理全部口径、调整责任部门、归档和发起全部重算。
- 运营部、财务部、供应链部非只读身份可以新增口径，并直接发布自己部门负责口径的新版本。
- 非总经办用户不能把责任部门改成其他部门，也不能修改或归档其他部门的口径。
- 其他获准查看数据中心的员工只读。
- 所有权限在服务端按认证会话校验；客户端不能提交操作者、发布时间或审计身份。

权限变更会扩展数据中心功能编辑部门，需要同步更新默认权限、产品角色文档和 API 权限契约。

## 共享 API

### 查询

- `GET /api/platform/v1/data-standards`：按分类、责任部门、状态和生效日期查询口径。
- `GET /api/platform/v1/data-standards/:id`：返回当前定义、版本历史、依赖和最近计算状态。
- `GET /api/platform/v1/data-standards/results`：按口径、时间和允许维度读取带版本及可信信息的结果。

### 写入

- `POST /api/platform/v1/data-standards`：新增并发布首个版本。
- `PUT /api/platform/v1/data-standards/:id`：基于 `expectedVersion` 发布新版本。
- `POST /api/platform/v1/data-standards/:id/archive`：归档口径。
- `POST /api/platform/v1/data-standards/preview`：校验公式并返回受限样本结果，不持久化结果。
- `POST /api/platform/v1/data-standards/recalculate`：建立历史重算任务。

错误响应使用稳定代码，至少区分认证、读取权限、责任部门写入权限、输入非法、未知字段、循环依赖、单位冲突、版本冲突、数据未覆盖、查询范围过大、计算失败和存储不可用。所有响应包含安全的 `requestId` 和可重试标识。

## D1 数据模型

### `data_metric_definitions`

保存口径身份、稳定 `metricCode`、分类、责任部门、单位、周期、当前版本、状态和归档信息。兼容迁移现有数据中心同名集合；不保存公式执行代码。

### `data_metric_definition_versions`

追加式保存版本号、名称、分类、责任部门、单位、周期、生效日期、展示公式、受控公式 AST、字段与指标依赖、数据源、创建人和时间。已发布版本不可更新；当前定义表只保存最新版本元数据用于列表和权限判断。

### `data_metric_results`

保存口径、版本、周期、允许维度、值、单位、覆盖率、可信等级、估算标识、数据截止时间、计算批次和生成时间。

### `data_metric_calculation_runs`

保存预览之外的计算或重算任务、范围、目标版本、状态、进度、错误码、发起人和时间。任务使用幂等键，重复提交不能生成重复有效结果。

### `data_metric_audit_logs`

追加式记录新增、发布、归档、预览失败、重算和结果切换，只记录必要字段名、版本和范围，不复制完整业务事实。

这些表需要进入 `docs/platform/environment-capabilities.json`、迁移清单、共享 API 文档和回滚说明。继续复用 `PRODUCT_FLOW_DB`，不新增变量、Secret 或绑定。

## 数据流与计算边界

1. 业务事实由现有销售、库存、采购、付款和盘点数据边界提供。
2. 口径服务按查询周期解析当时生效的版本并编译受控公式。
3. 执行器读取白名单事实，完成聚合、过滤和依赖指标计算。
4. 结果连同覆盖率、可信等级、估算标识和版本写入结果表。
5. 数据中心总览和其他 App 只读取口径结果，不在 React 组件中复制公式。

销售五项首批口径接入后，数据中心现有净销售额、毛利、销量、退款率和毛利率全部改从共享计算结果读取。没有已发布口径或结果不完整时，卡片显示原因，不回退到另一套硬编码公式。

## 页面设计

- 左侧导航和页面标题统一使用“数据口径”，保留 `data-metrics` 路由兼容历史链接。
- 页面顶部展示治理规则、新增口径按钮、分类筛选、状态筛选和搜索。
- 销售与货流分区展示口径名、`metricCode`、当前公式、生效版本、覆盖状态、责任部门和操作。
- 详情展示公式结构、来源字段、依赖图、版本历史、最近结果和影响的订阅 App。
- 新增/编辑使用分步构建器：基本信息、数据范围、公式、样本预览、发布确认。
- 归档前展示依赖；重算前展示范围和影响；危险操作使用明确确认文案。
- 覆盖加载、空、错误、无权限、版本冲突、公式非法、数据未覆盖、计算中、重算失败、键盘、焦点、窄屏和钉钉 WebView 状态。

## 兼容、迁移与回滚

1. 迁移现有净销售额、毛利定义为共享口径 `v1`，再写入其余九项系统首批定义。
2. 首批货流事实未覆盖时只发布定义，不生成结果。
3. 新总览切换前同时运行旧计算和新引擎进行只读对账；差异未解释前不切换读取源。
4. 切换使用可回滚的读取开关；回滚只恢复旧读取，不删除新表、版本、结果或审计。
5. 数据库迁移必须向后兼容，提供建表回滚说明，但生产回滚不物理删除已产生的治理证据。
6. 本地、Preview、Production 分别验证；未获授权不执行生产迁移、写入、部署或真实外部平台操作。

## 验收标准

1. 左侧及页面标题使用“数据口径”，路由仍为 `data-metrics`。
2. 首批展示五项销售和六项货流口径，无重复 `id` 或 `metricCode`。
3. 授权部门可以新增、发布自己部门的新版本并归档；越权写入由服务端返回 403。
4. 公式构建器不能提交任意代码，并阻止未知字段、循环依赖、单位冲突和除零缺陷。
5. 修改生成带生效日期的新版本，历史查询解析当时版本，不静默重算。
6. 归档不删除历史；存在依赖时页面展示影响范围。
7. 显式重算生成新批次，失败不替换当前有效结果，重复提交保持幂等。
8. 数据中心五项销售 KPI 全部读取共享口径结果；无有效结果时显示缺失原因，不回退硬编码算法。
9. 每个结果包含版本、覆盖率、可信等级、估算标识和数据截止时间。
10. 环境能力清单、API 目录、迁移、权限文档、错误契约和测试同步更新。
11. 完成领域、API、权限、兼容、公式、重算、UI、响应式和项目完整 Definition of Done 验证。
