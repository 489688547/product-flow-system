# 抖音店铺自动登录识别设计

## 状态

- 日期：2026-07-20
- 状态：已确认，实施中
- 负责人：总经办 / 运营数据管理员
- 关联平台：抖音电商、Cloudflare Pages、Cloudflare D1、公司 Mac 浏览器助手

## 问题与目标

当前抖音连接弹窗要求用户填写店铺名称、账户类型、后台地址、登录手机号、同步数据等信息。这些信息大多可以由平台定义或登录结果确定，重复填写会增加误差。

目标流程只要求用户填写登录邮箱和密码。系统加密保存凭证后，在公司 Mac 上打开固定后台地址 `https://fxg.jinritemai.com/login/common?channel=zhaoshang`，完成登录并自动识别店铺名称和店铺头像。数据接入列表最终只展示店铺头像和店铺名称。

## 已确认产品规则

1. 抖音连接弹窗只展示“登录邮箱”和“密码”两个业务字段。
2. 后台地址由服务端平台定义固定，用户不可查看或修改技术配置。
3. 邮箱保存后直接回显。
4. 密码默认掩码；有连接管理权限的用户点击“小眼睛”后，可在当前页面内短时查看明文。
5. 明文不写入浏览器存储、URL、日志、审计或普通列表响应；关闭弹窗、页面失焦或 60 秒后清空。
6. 保存成功后创建登录验证任务；公司 Mac 浏览器助手自动打开固定地址并填写邮箱和密码。
7. 验证码、扫码、滑块、短信或设备确认必须由人处理，系统不保存这些一次性信息。
8. 登录成功后，浏览器助手读取店铺 ID、店铺名称和店铺头像地址并回写；连接状态才从“等待登录确认”变为“已连接”。
9. 数据接入列表不再展示接入方式、账户类型、后台地址、同步数据、负责人或技术状态说明；每个已识别实例只显示店铺头像、店铺名称和简短连接状态。
10. 同一个抖音登录账号可以产生一个或多个店铺实例；以平台店铺 ID 去重，不以用户手填名称去重。

## 方案比较

### 方案 A：公司 Mac 浏览器助手（采用）

浏览器助手领取一次性验证任务，使用公司 Chrome 的独立工作配置打开抖音后台、填入凭证、等待人工验证并读取店铺信息。

优点：能复用真实浏览器环境和设备信任；验证码可自然转人工；后续可直接承接每日采集。缺点：公司 Mac 必须在线，页面结构变化时需要更新识别规则。

### 方案 B：Chrome 扩展

由扩展监听抖音后台页面并回传店铺信息。

优点：与浏览器结合紧密。缺点：需要安装、升级和较高网页权限；组织分发成本高。首期不采用，可作为浏览器助手的实现载体之一。

### 方案 C：纯 AI 视觉登录

持续截屏并让视觉模型决定点击位置和识别文字。

优点：对部分页面改版更有弹性。缺点：不稳定、成本高、延迟大，并可能扩大敏感画面外发范围。不得作为主链路。

## 能力边界决定

决定为“扩展现有能力”。凭证加密、版本、权限和审计复用现有平台连接保险箱及 AES-GCM 能力，不创建第二套加密协议。公司已有用户洞察浏览器采集器，证明浏览器任务、设备身份和任务回传是第二个真实消费者；因此把设备认证和任务领取抽到共享浏览器助手边界，抖音页面识别仍保留在 provider adapter 内。

### 通用数据采集平台边界

抖音店铺识别不是独立自动化脚本，而是共享数据采集平台的第一个 provider adapter。平台统一负责连接实例、加密凭证、设备身份、定时或人工触发任务、一次性 task grant、人工验证接管、重试、游标、结果回写和审计；每个平台 adapter 只负责登录字段定义、允许访问的固定域名、可采集资源、页面/API 操作和标准化映射。

统一任务契约使用 `providerId`、`taskType`、`resourceType`、`connectionId`、`schemaVersion` 和 `cursor`。抖音首个资源类型为 `connection_identity`，结果是店铺 ID、名称和头像；后续 ERP、快手、天猫、巨量等接入订单、商品、库存、广告消耗或素材时复用相同任务和设备协议，但由各自服务端 ingestor 写入相应标准事实表，不把所有异构明细塞进一个万能 JSON 表。

provider registry 默认拒绝未登记的 provider、域名、任务类型和资源类型。首期只启用 `douyin-ecommerce/connection_identity`；ERP 等平台先拥有稳定扩展接口，不在没有字段口径和真实页面验证时伪装为已接通。

## 界面设计

### 数据接入列表

- 未配置：抖音平台 Logo、`抖音电商`、`添加店铺`。
- 等待验证：抖音平台 Logo、登录邮箱脱敏摘要、`等待登录确认`。
- 已连接：店铺头像、店铺名称、`已连接`；点击整行进入管理弹窗。
- 头像缺失或失效：回退抖音平台 Logo，不出现破图。
- 多店铺：逐店铺展示，不在一个卡片里堆叠技术字段。

### 抖音专属弹窗

只包含：

- 登录邮箱：新建时必填；已保存时回显。
- 密码：新建时必填；已保存时显示掩码。点击小眼睛触发受控 reveal，60 秒后自动隐藏。
- 主按钮：新建为“保存并打开登录”，已保存为“保存并重新验证”。
- 次按钮：取消。

弹窗不显示店铺名称、账户类型、后台地址、登录手机号、接入方式、同步数据、负责人、数据集或加密实现说明。登录验证的进度在提交后用四个业务状态表达：`正在等待公司 Mac`、`请在抖音页面完成人工验证`、`正在识别店铺`、`已识别店铺名称`。

## 数据与 API

### 连接实例

每个抖音连接实例保存非敏感元数据：

```json
{
  "id": "connection-id",
  "platformId": "douyin-ecommerce",
  "accountLabel": "operator@example.com",
  "credentialSchemaId": "email-password-v1",
  "loginEmail": "operator@example.com",
  "shopId": "platform-shop-id",
  "shopName": "店铺名称",
  "shopAvatarUrl": "https://...",
  "status": "waiting_verification",
  "credentialVersion": 1,
  "lastVerifiedAt": "2026-07-20T00:00:00.000Z",
  "version": 1
}
```

底层连接表使用通用 `accountLabel`、版本化 `credentialSchemaId` 和共享保险箱引用，不使用抖音专属邮箱列或第二份密文。密码只存在 `credential_vault_entries` 的加密 JSON 中。`loginEmail` 是抖音 API/UI 的兼容字段，按用户确认属于可回显的账户元数据，但不写入公开集成注册表和日志。

### 浏览器任务

任务字段固定为平台、连接实例、目标动作、状态、过期时间、领取设备、尝试次数和结果摘要。任务不保存明文密码。公司 Mac 使用设备 Bearer Token 领取任务，再凭一次性、短时、绑定连接实例的 task grant 获取当前任务所需凭证。grant 使用一次后失效，最长五分钟。

### 路由

- `GET|POST|PUT /api/platform/v1/data-connections`：读取、创建和更新连接实例。
- `POST /api/platform/v1/data-connections/:id/reveal`：受控查看邮箱和密码。
- `GET /api/platform/v1/browser-agent/tasks`：公司 Mac 领取任务。
- `POST /api/platform/v1/browser-agent/tasks/:id/result`：回写等待人工、成功或失败结果。

所有用户路由使用公司钉钉会话；管理与 reveal 权限由服务端校验。设备路由使用独立设备令牌和 task grant，不接受浏览器公司的普通会话替代。

## 公司 Mac 执行流程

1. 浏览器助手领取 `douyin_login_verification` 任务。
2. 获取一次性凭证 grant，在内存中解密邮箱和密码。
3. 使用独立 Chrome Profile 打开固定抖音后台地址。
4. 优先用稳定 DOM 选择器填写登录邮箱和密码并提交。
5. 遇到验证码、滑块、扫码、短信或设备确认时暂停并回报 `waiting_human_verification`。
6. 人工完成后继续等待已登录页面。
7. 先读取稳定页面数据或 DOM 中的店铺 ID、店铺名称和头像地址。
8. 识别成功后回写，清空内存凭证并关闭本次 task grant。
9. 登录失败、超时或页面结构变化时保留旧店铺信息，并返回可重试错误，不把连接误标记为已连接。

## AI 辅助识别

主链路使用确定性 DOM 规则。只有检测到页面结构变化且未包含登录表单、密码、验证码或个人消息时，才允许截取登录成功后的店铺信息区域交给已配置的公司 AI Provider。模型只返回候选店铺名、候选头像区域和置信度。

- 置信度达到 0.90 且店铺 ID 可由页面规则独立确认时，可自动回写。
- 其他情况进入人工确认，不自动覆盖现有店铺名称。
- AI 请求不包含邮箱、密码、Cookie、Token、完整页面 HTML 或验证码。

## 权限与明文查看

- 总经办非只读和被授权的运营数据管理员可以创建、替换、重试和查看自己管理范围内的抖音连接。
- reveal 要求当前登录会话创建不超过 15 分钟、明确点击小眼睛，并写入追加式审计。
- reveal 响应使用 `Cache-Control: no-store, private`、`Pragma: no-cache` 和 `X-Content-Type-Options: nosniff`。
- 同一连接 15 分钟内最多成功 reveal 5 次；超过后返回稳定限流错误。
- 列表、普通 GET、错误响应和审计永远不返回密码。

## 错误与恢复

- 公司 Mac 离线：保持凭证已保存，状态为“正在等待公司 Mac”，可重试。
- 人工验证未完成：状态为“请在抖音页面完成人工验证”，任务不过早失败。
- 凭证错误：提示重新填写，不回显服务端底层错误。
- 页面结构变化：确定性规则停止写回；可进入 AI 辅助或人工确认。
- 店铺头像失效：列表回退平台 Logo，后续验证刷新头像地址。
- 版本冲突：保留本地草稿，刷新后再次提交。
- 重复任务：同一连接同一凭证版本只允许一个活动验证任务，使用幂等键复用。

## 迁移、兼容和回滚

- 新连接实例和浏览器任务使用新增 D1 表；密码复用现有数据中心凭据保险箱，不写入连接元数据表。
- 线上已有的抖音连接器元数据按邮箱和凭证关联迁入；缺少邮箱或凭证的重复记录保持只读“待整理”，不得猜测合并。
- 现有平台连接加密主密钥、算法和密钥版本继续使用。
- 回滚时停用新页面和任务创建，保留加密凭证、连接实例、店铺识别结果和审计，不物理删除。

## 验收与验证

1. 抖音弹窗只有邮箱和密码两个业务字段。
2. 固定后台地址不出现在可编辑表单和请求体。
3. 邮箱可回显；密码默认掩码，受控 reveal 后可见并在 60 秒内清空。
4. 保存后产生一个幂等登录验证任务，公司 Mac 能领取并打开固定地址。
5. 验证码等一次性信息不会写入 D1、日志或 API。
6. 登录成功后按店铺 ID 回写名称和头像；外层只展示店铺头像、店铺名称和连接状态。
7. 多店铺不会因同名或同邮箱互相覆盖。
8. Mac 离线、人工验证、错误凭证、页面改版、AI 低置信度和版本冲突均有明确恢复路径。
9. 匿名、只读、越权运营和伪造设备任务均被服务端拒绝。
10. 1440、900、640、390px 以及钉钉 WebView 无页面级横向溢出，键盘、焦点、Esc 和明文清除行为通过。
