# 外部平台集成路由 PRD

## 问题

平台知识分散，后续开发无法稳定判断钉钉、快麦、阿里云、Cloudflare 等系统是否相关，容易遗漏官方能力、账号边界和代码影响。

## 用户

- 公司员工：在说明书查看平台能力、状态、关系和内部使用入口。
- 开发者与 Codex：开发前识别相关平台和必读资料。
- 总经办平台管理员：维护内部平台资料并接受审计。

## 目标与业务规则

- 公开注册表是平台状态、能力、关键词、代码边界和关系的唯一事实源。
- 所有平台使用已连接、集成中、计划中、已停用四种状态。
- 路径命中必须在 PR 声明平台影响，文本关键词只做建议。
- 所有登录员工可读内部资料，仅总经办可编辑。
- 不保存任何密钥、令牌、Cookie、密码或私钥。
- 内部 API 故障时公开平台地图仍可使用。

## 非目标

- 不在本功能内完成新平台 API 接入。
- 不把说明书变成凭据管理器。
- 不绕过平台官方权限和现有应用认证。

## 验收标准

1. 开发者可通过任务文本或变更路径得到平台候选、依据和关系平台。
2. PR 路径命中平台但声明缺失或漏报时，`quality` 检查失败。
3. 员工可从说明书搜索并筛选全部初始平台。
4. 登录员工能看到内部覆盖资料；API 失败时仍看到公开资料和降级提示。
5. 总经办可编辑内部资料，普通员工写入被服务端拒绝。
6. 注册表和内部 API 均拒绝敏感字段。
