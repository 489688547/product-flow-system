# 用户洞察功能设计书

## 设计目标

让运营和产品在不离开统一工作台的情况下，看清“哪个平台、哪个店铺或产品、哪个类目、哪一批数据”支撑当前判断。页面优先呈现可追溯事实，再呈现由本部门规则产生的参考建议；平台差异、采集缺口和人工确认状态始终可见。

## 信息架构

数据中心左侧导航新增“用户洞察”，路由段为 `data-insights`。页面采用已确认的单工作区布局，不增加第二套侧栏：

1. 页头：标题、说明、数据截止时间、刷新与手动重试。
2. 视角切换：`店铺视角` / `产品/SKU 视角`。
3. 范围栏：平台、店铺或产品、SKU、平台类目、时间范围。
4. 页内 Tab：`用户分析` / `竞品分析` / `规则与建议`。
5. 主工作区：当前 Tab 的事实、对比、证据或规则。
6. 固定说明：`洞察与建议仅供参考，不会自动修改经营数据或平台设置。`

视角和平台是页面的最高优先级筛选。用户不会在同一屏被迫同时阅读三类工作区；切换页内 Tab 时保留已选范围，切换店铺与产品视角时分别恢复各自最近一次有效范围。

## 页面布局

### 页头与范围栏

页头左侧显示“用户洞察”和一句用途说明，右侧显示最后成功时间、当前质量状态、“刷新数据”和“手动重试”。刷新只读取数据中心已保存事实；手动重试仅在本地采集器在线、类目已确认且用户有权限时可用。

范围栏顺序固定为：

- 视角切换；
- 平台单选；
- 店铺单选，或产品与 SKU 级联选择；
- 平台类目选择与确认状态；
- 数据范围。

平台类目控件显示“系统推荐”“待确认”“已确认”或“已停用”。待确认时提供推荐依据和“确认类目”；无权限用户看到只读原因。选择平台后不自动套用其他平台的类目。

### 用户分析

主区按信息价值排列：

1. 摘要带：目标人群、需求/兴趣变化、类目规模或热度、数据覆盖率与新鲜度。
2. 用户事实：平台原生人群分层、地域/年龄/兴趣等已提供维度；不支持项不占位造数。
3. 市场内容切换：商品、视频、直播三个子视图，展示主卖款、热度、价格、内容表现或直播表现等平台可验证字段。
4. 差异与证据：当前结果相对上一窗口或所选对照范围的变化，以及来源页面、采集时间和质量。

跨平台模式不绘制一条虚假的总趋势。共同口径使用分平台列或小倍图；平台特有指标留在各平台区块，并说明不能直接比较。

### 竞品分析

顶部区分两个列表：

- `核心竞品`：人工维护或已确认候选，可进入正式对比。
- `候选竞品`：由当前 App 的规则发现，显示命中条件、来源维度、证据和推荐原因。

候选行的操作为“确认加入”“驳回”“查看证据”。确认必须填写或选择原因，记录当前规则版本；驳回可选择暂不相关、非直接竞品、数据异常或自定义原因。只读用户没有操作按钮。

正式对比按平台展示商品、价格、内容、视频、直播和用户差异。缺少的维度显示“该平台暂无可验证数据”，不以 0 参与评分。任何综合提示都展示组成指标和权重，不显示无法解释的单一分数。

### 规则与建议

该 Tab 分为“当前规则集”“建议列表”“规则版本”三段：

- 当前规则集显示消费 App、负责人部门、平台、适用范围、竞品条件、指标权重、最低覆盖率和版本。
- 建议列表按用户、市场、产品、运营分类，卡片包含条件性结论、证据、置信度、覆盖率、新鲜度和限制。
- 规则版本展示创建、复制、发布、停用记录，并可查看差异。

“复制其他 App 规则”先打开只读预览，确认后创建归属当前 App 的草稿版本。编辑和发布均使用版本校验；发生并发冲突时保留本地草稿，要求刷新后重新合并。

## 关键用户故事

### 产品负责人确认类目并查看用户需求

1. 从产品/SKU 视角选择产品和平台。
2. 系统展示推荐类目路径及依据。
3. 产品负责人确认类目，采集计划从下一工作日生效，也可在采集器在线时手动重试。
4. 用户分析展示该平台类目用户、商品、视频和直播事实。
5. 产品负责人打开产品全周期规则产生的建议，查看证据后自行决定是否进入产品规划。

### 运营确认候选竞品

1. 从店铺视角选择平台、店铺和已确认类目。
2. 打开竞品分析查看运营规则发现的候选。
3. 查看候选命中的主卖款、价格、视频或直播规则及来源。
4. 确认后候选进入当前运营规则集的核心竞品；产品规则集不受影响。
5. 后续建议基于核心竞品和当前版本生成，仅作为运营方案参考。

### 复制另一个 App 的规则

1. 用户查看有权限读取的产品或运营规则。
2. 选择“复制为本 App 规则”，系统展示来源、版本和将被替换的范围引用。
3. 保存后生成新的规则 ID 和草稿版本，源规则不变。
4. 用户按本部门口径修改并发布；候选与建议标明新规则版本。

### 采集登录失效

1. 采集器发现页面进入登录、验证码或风控状态。
2. 当前来源立即停止，批次记为 `login_required`，不读取登录页内容。
3. 页面继续显示最后成功快照、截止时间和受影响维度。
4. 授权用户完成登录后点击手动重试；新批次独立记录，不覆盖失败历史。

## 浏览器采集设计

### 运行边界

本地采集器运行在公司 Mac，复用用户已经登录的 Chrome 会话。采集范围来自数据中心登记的 HTTPS 后台地址、允许域名、店铺标签、已确认类目和市场页类型。采集器不得枚举浏览历史、扫描其他标签页或读取浏览器凭证存储。

工作日按数据中心截止时间执行；周末跳过自动任务。人工重试必须由有权限的用户发起，并写入操作者、原因、来源范围和结果。

### 平台适配器

每个平台使用独立适配器，并显式声明：

- 支持的市场入口和域名；
- 类目选择方法与页面结构版本；
- 支持的用户、商品、视频和直播维度；
- 优先导出格式及回退页面表格/请求响应；
- 字段映射、单位、时间口径和缺失处理；
- 登录、风控、结构变化和不支持判定；
- 固定夹具和生产验收记录。

适配器只将可验证字段映射到标准模型。平台特有字段保存在有命名空间的扩展字段中，不通过猜测转成通用指标。

### 批次与写入

采集器为每个平台、店铺、类目、维度和时间范围建立批次，计算内容哈希并上传结构化记录。服务端先校验设备令牌、允许范围、结构版本、记录数量和幂等键，再一次性完成标准事实写入。任何批次失败都不替换最后成功快照。

原始结构化快照保留在本机受控目录，用于故障复核和重新解析；服务端只保存来源引用、标准事实、覆盖率、结构版本、内容哈希及同步日志。

## 共享平台能力

### 抽取结论

共享抽取是必要的，因为数据中心、产品全周期和电商店铺运营已有三个真实消费者。共享范围包括：

- 用户洞察查询和范围授权；
- 平台类目推荐与确认；
- 浏览器采集注册、批次、幂等和错误契约；
- 标准市场事实与平台扩展字段；
- App 归属的规则版本；
- 候选竞品确认和审计；
- 参考建议的证据、质量与限制。

这些能力通过 `src/domain` 的纯规则、`src/state` 的客户端和 `/api/platform/v1/user-insights...` 服务端契约提供。外部平台适配只存在于服务端或本地采集器边界，页面不得直接调用平台。

### 保持局部的能力

用户洞察页面的摘要带、筛选组合、候选列表和规则编辑器先保留在 `src/features/data-center`。当出现第二个真实界面消费者后，才把业务中立的基础控件抽到 `src/ui`。

## 交互状态

### 加载

保留页头、范围栏和当前 Tab 布局，使用三段骨架；加载期间不清空用户当前筛选。切换平台时旧平台数据降低视觉强调，并标明正在加载新平台，避免误读。

### 空数据

按原因提供不同动作：未选范围时引导选择；类目未确认时引导确认；尚未采集时显示下次工作日计划和手动重试条件；平台不支持时说明限制。空状态不展示虚假图表。

### 部分数据

保留可验证数据，在摘要和受影响模块显示覆盖率。缺失指标使用 `—`，建议降级为条件性表达，禁用依赖完整数据的确认动作并解释原因。

### 过期

保留最后成功快照，页头和相关卡片显示数据截止时间、过期原因和受影响维度。刷新只重新读缓存事实；采集器可用时才显示手动重试。

### 错误

- 需要登录：显示“请在已登记平台页面完成登录后重试”。
- 页面结构变化：显示“采集规则需要更新”，不提供强制继续。
- 批次失败：显示安全错误摘要和批次编号，不展示原始响应。
- 权限不足：页面只读或 API 返回 403，并解释可联系的负责部门。

### 成功

确认类目、发布规则、确认竞品或手动重试后显示对象、版本、操作者和时间。成功提示不自动消失到无法核对；活动记录可再次查看。

## 组件策略

复用现有 `PageHeader`、`Button`、`DataTable`、状态标签、表单、骨架和错误提示规则。新增局部组件建议包括：

- `InsightScopeBar`：视角、平台、对象、类目和日期组合；
- `InsightEvidenceSummary`：来源、范围、覆盖率和新鲜度；
- `MarketDimensionWorkspace`：用户、商品、视频和直播事实组合；
- `CompetitorCandidateList`：候选证据与确认动作；
- `InsightRuleWorkspace`：App 归属规则、版本和复制流程；
- `AdvisorySuggestionCard`：建议、证据、限制与“仅供参考”。

这些组件暂为功能局部组件，不包含 API 请求；数据由 Provider/state 层注入。领域归一化、覆盖率、候选状态和建议降级判断不得写在 React 组件中。

## 响应式与钉钉 WebView

- 1440–1280px：范围栏保持一行或自然分组，主工作区使用事实主区和证据辅助区。
- 900px：范围栏分两行，主辅区改为一列，表格在容器内滚动。
- 640px：视角、平台、对象和类目改为纵向；页内 Tab 保持可滚动但具有完整标签。
- 390px：无页面级横向溢出；表格、竞品指标和版本差异只在自身容器横向滚动。
- 钉钉 WebView 使用动态视口与安全区；下拉层不被底部导航遮挡，软键盘出现时保存/确认按钮仍可到达。

## 无障碍

- 视角和 Tab 使用正确的单选或 tab 语义，支持方向键并设置 `aria-selected`。
- 所有筛选器、规则字段和确认原因有可关联 label、错误提示和帮助说明。
- 状态、覆盖率和置信度同时使用文字，不只依赖颜色。
- 图表必须提供文字摘要和表格等价信息。
- 候选确认、驳回、规则发布等高影响操作支持键盘并在完成后把焦点移到结果提示。
- 动画遵循 `prefers-reduced-motion`，对比度达到 WCAG AA。

## 数据与 API 设计约束

稳定共享接口使用以下资源方向，具体字段、认证和兼容策略在实施计划及平台 API 契约中固化：

- `GET /api/platform/v1/user-insights`：按授权范围查询事实、质量和建议；
- `/api/platform/v1/user-insight-category-mappings`：推荐、确认和停用类目；
- `/api/platform/v1/user-insight-rules`：规则草稿、复制、发布、版本和停用；
- `/api/platform/v1/user-insight-competitors`：核心竞品、候选、确认和驳回；
- `/api/platform/v1/user-insight-ingest`：设备注册范围内的批次开始、分片、完成和失败。

持久化采用新增表保存规则、类目映射、快照、标准实体、竞品、同步运行、设备令牌哈希和审计。迁移只新增结构，并提供关闭读取、停止采集和保留数据的回滚路径。

## 测试与视觉验收

### 自动测试

- 领域：平台可比字段、覆盖率、建议降级、类目状态、候选状态、规则复制和版本冲突。
- 解析：每个平台固定的导出、页面或请求夹具；登录页、字段缺失、结构变化和单位转换。
- API：认证、组织/App/范围授权、设备令牌、幂等、超时、失败、兼容和审计。
- React：两个视角、平台切换、三个页内 Tab、类目确认、候选确认、规则复制及所有空错禁用状态。
- 集成：采集夹具到标准事实、事实到建议、产品/运营 App 只读消费契约。

CI 不打开真实平台或使用真实账号。生产验收作为独立检查，只在公司 Mac 已授权和已登录页面执行。

### 视觉验收矩阵

检查 1440、1280、900、640 和 390px 下的以下状态：

- 店铺与产品/SKU 视角；
- 用户、竞品、规则与建议 Tab；
- 已确认、待确认和无类目；
- 有数据、空数据、部分、过期、需要登录、结构变化、失败和只读；
- 单平台、跨平台共同字段对照和平台特有字段；
- 键盘全流程、焦点返回、钉钉 WebView 与无页面级横向溢出。
