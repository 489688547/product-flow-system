# 用户洞察 PRD

## 文档状态

- 状态：待书面复核
- 负责人：数据中心 / 产品部 / 运营部
- 最近更新：2026-07-20

## 背景与问题

不同电商平台的主卖款、用户结构、内容形态和竞争环境并不相同。运营和产品目前需要分别进入各平台后台的“市场”区域，选择类目后查看用户、商品、视频和直播数据，再把零散信息人工整理成产品判断或运营建议。这一过程缺少统一的数据口径、采集记录、竞品确认规则和证据链，也容易把一个平台的结论直接套用到另一个平台。

数据中心已经承担店铺经营事实、来源、质量和跨 App 服务。新增“用户洞察”应继续由数据中心管理采集事实和标准化结果，同时允许产品全周期与电商店铺运营 App 分别定义本部门的分析规则。系统只提供参考建议，不代替产品或运营负责人确认类目、竞品和经营动作。

## 目标

1. 在数据中心新增一级“用户洞察”Tab，提供店铺视角和产品/SKU 视角。
2. 在每个平台内分别展示用户分析、竞品分析及规则与建议，保留平台差异，不把不可比指标强行合并。
3. 通过公司 Mac 上的浏览器采集器读取已登录平台后台的市场/类目页面，覆盖用户、商品、视频和直播四类市场事实。
4. 由系统推荐平台类目，由产品或运营人员确认后才进入采集范围。
5. 支持人工维护核心竞品，并按部门规则发现候选竞品；候选必须经人工确认后才能进入正式对比。
6. 让产品全周期和电商店铺运营 App 各自维护版本化规则，同时复用数据中心的标准事实、查询能力和建议证据。
7. 所有建议标明数据范围、更新时间、覆盖率、置信度和“仅供参考”，不得自动修改经营数据或执行外部动作。

## 非目标

- 不通过官方开放 API、第三方爬虫服务或账号密码登录平台。
- 不读取或保存密码、Cookie、Token、验证码、浏览历史、无关标签页或个人数据。
- 不自动关注竞品、发送消息、调整价格、预算、广告计划、商品设置、库存或内容排期。
- 不把系统发现的候选竞品直接认定为核心竞品。
- 不把跨平台不可比较的指标做伪合计、伪排名或统一画像。
- 不根据洞察自动评价员工、产品等级或运营绩效。
- 不把缺失字段、采集失败或未登录状态当作 0 或正常数据。

## 用户与权限

### 总经办

查看全部平台、店铺、产品和部门规则；维护全局安全与采集边界；必要时确认或撤销跨部门可见范围。不得绕过来源范围执行平台动作。

### 产品部

查看被授权的用户与竞品事实；确认产品与平台类目的映射；维护产品全周期 App 的用户需求、产品机会、竞品筛选和建议规则；确认该规则集产生的候选竞品。可查看或复制运营规则，但不能覆盖运营规则。

### 运营部

查看被授权的平台、店铺和产品洞察；确认运营范围内的类目映射；维护电商店铺运营 App 的平台、价格、内容、直播、主卖款和竞品规则；确认该规则集产生的候选竞品。可查看或复制产品规则，但不能覆盖产品规则。

### 其他部门

默认不显示用户洞察入口。后续如需消费事实，必须通过明确的 App 权限和数据服务契约接入。

### 服务身份

浏览器采集器使用哈希后的设备令牌调用限定范围的采集入口。令牌只允许写入已登记平台、店铺、确认类目、数据维度和时间范围，不能获得普通用户查询或管理权限。

## 当前流程

1. 产品或运营人员进入不同平台后台寻找市场数据。
2. 人工选择类目，分别查看用户、商品、视频和直播榜单或趋势。
3. 人工挑选竞品并在文档中记录判断，规则与证据无法稳定复用。
4. 不同部门分别整理同一批事实，跨平台数据容易混用或丢失更新时间。
5. 页面登录失效、字段变化或数据不完整时，常以经验继续判断，缺少明确的数据质量状态。

## 目标流程

1. 用户进入数据中心“用户洞察”，选择店铺视角或产品/SKU 视角。
2. 用户选择平台、店铺或产品；系统根据现有映射和后台信息推荐平台类目。
3. 产品或运营人员确认类目后，数据中心将该平台市场页加入允许采集范围。
4. 工作日到达数据中心设定的采集截止时间后，本地采集器复用已登录 Chrome 会话，逐个读取已登记页面。
5. 采集器优先读取平台导出；无导出时读取已确认的页面表格或请求响应，并提交结构化批次、覆盖率和页面结构版本。
6. 数据中心校验身份、来源范围、幂等键和字段版本，保存标准化用户、商品、视频、直播事实及同步状态。
7. 页面按平台展示用户分析；竞品页同时展示人工核心竞品和规则发现的候选竞品。
8. 负责人确认或驳回候选，系统依据当前 App 的规则版本生成带证据的参考建议。
9. 产品全周期和电商店铺运营 App 通过共享查询契约读取事实与建议，分别形成产品或运营判断。

## 业务规则

1. 数据中心 App 新增 `data-insights` 一级入口，名称为“用户洞察”。
2. 页面默认提供“店铺视角”和“产品/SKU 视角”，二者共享数据事实但不共享筛选状态。
3. 每次分析必须选择一个平台；跨平台总览只展示共同口径、平台分列结果和差异说明，不合并不可比指标。
4. 平台类目映射状态为“系统推荐、待确认、已确认、已停用”。只有“已确认”且当前用户有权维护的映射可进入采集计划。
5. 类目推荐必须展示推荐依据和来源路径；人工确认保存确认人、确认时间和版本。
6. 采集仅在工作日自动运行，默认遵循数据中心现有 `07:30` 截止设置；周末不自动运行，允许授权用户手动重试。
7. 采集器只访问已登记平台域名、后台入口、店铺账户标签和已确认类目页面；任何新页面必须先登记再采集。
8. 原始页面结构化快照保存在本地受控归档；D1 只保存标准事实、实体引用、摘要、质量、同步和审计信息，不保存完整 HTML、截图、Cookie 或凭证。
9. 每批次使用平台、店铺、类目、维度、数据范围、结构版本和内容哈希形成幂等键；重复批次不得重复写入事实。
10. 洞察维度固定为用户、商品、视频和直播。平台缺少某维度时显示“不支持”或“无数据”，不能补造。
11. 核心竞品由人维护。系统发现的实体初始只能是“候选”，必须展示命中规则和来源证据；人工确认后才成为当前规则集的核心竞品。
12. 竞品状态为候选、核心、已驳回和已停用；确认、驳回、恢复和停用均保存操作者、原因和时间。
13. 规则集必须归属一个消费 App、负责人部门、平台和适用范围，并保存版本。产品全周期规则与电商店铺运营规则相互独立。
14. 用户可以查看另一个 App 的已授权规则并“复制为新规则”；复制产生本 App 的新 ID 和新版本，不能覆盖源规则。
15. 规则可限定店铺、产品、SKU、已确认类目、竞品条件、指标权重、观察窗口和最低覆盖率。系统不得跨越规则范围推断。
16. 建议只分为用户、市场、产品和运营四类，必须引用事实、规则版本、数据范围、覆盖率、置信度和新鲜度。
17. 覆盖率不足、数据已过期、字段缺失或只有单平台小样本时，建议必须降级为条件性表达；不输出确定归因。
18. 所有建议和详情页固定展示“仅供参考”，不能自动改变产品规划、运营方案、竞品状态或外部平台配置。
19. 页面和 API 均按组织身份、消费 App、平台、店铺和产品范围授权；隐藏控件不是权限边界。

## 数据定义

### `InsightScope`

一次查询或规则的适用范围：`viewType`（`shop` 或 `product`）、平台、店铺、产品、SKU、已确认类目、时间范围和数据维度。

### `InsightRuleSet`

规则集 ID、消费 App ID、负责人部门、平台、适用范围、竞品发现条件、指标权重、最低覆盖率、状态、版本、创建人、更新时间和来源规则引用。

### `CategoryMapping`

平台、来源类目 ID、类目名称及路径、店铺或产品引用、系统推荐依据、状态、确认人、确认时间和版本。

### `MarketSnapshot`

平台、店铺、类目、维度（用户/商品/视频/直播）、数据范围、采集时间、来源页面引用、页面结构版本、内容哈希、覆盖率、质量状态和标准化指标。

### `MarketEntity`

标准化用户分群、商品、视频或直播实体。保存平台实体 ID、名称、平台内指标、所属快照和可比较字段；平台特有字段保存在显式命名的扩展结构中。

### `CompetitorCandidate`

规则集、平台实体、命中规则、推荐理由、候选/核心/已驳回/已停用状态、确认人、确认原因和确认时间。

### `InsightSuggestion`

建议类别、标题、条件性结论、证据引用、规则版本、数据范围、覆盖率、置信度、新鲜度、限制说明、生成时间和 `advisoryOnly=true`。

### `InsightSyncRun`

采集器、平台、店铺、类目、维度、计划时间、开始/结束时间、状态、读取行数、写入行数、内容哈希、错误代码、结构版本和最后成功快照。

## 共享能力边界

用户洞察事实、类目映射、浏览器采集契约、规则版本、候选竞品和建议查询会被数据中心、产品全周期和电商店铺运营三个真实消费者使用，因此属于共享平台能力。稳定接口使用 `/api/platform/v1/user-insights...` 命名空间，并在平台 API 契约中记录认证、授权、请求、响应、错误、兼容、幂等和审计。

数据中心页面的筛选布局、洞察卡片和工作区组合只服务本页面，保留在 `src/features/data-center`，不因预测性复用进入共享 UI。产品和运营 App 只消费共享数据契约，不直接访问平台后台或浏览器采集器。

## 异常与边界

- `login_required`：采集页面进入登录、验证码或风控状态时立即停止该来源，提示人工完成登录后重试；不尝试绕过。
- `schema_changed`：导出列、页面表格或请求结构与已登记版本不一致时停止写入，保留上次成功快照并标记需要更新采集规则。
- `partial`：部分维度或字段缺失时保存可验证数据和覆盖率，不把缺失值写成 0，不生成高置信度建议。
- `stale`：没有达到数据中心新鲜度要求时继续展示上次成功结果，并突出截止时间和影响范围。
- `failed`：网络、批次校验或存储失败时不覆盖最后成功快照，提供安全错误摘要和手动重试。
- `unsupported`：平台没有对应市场维度或无法以允许方式读取时明确显示不支持。
- 映射未确认：允许查看推荐，不进入自动采集，不生成依赖该类目的正式建议。
- 规则并发：服务端按版本拒绝覆盖，要求刷新后复制或合并。
- 权限不足：服务端返回 403；已有快照不能因前端缓存绕过范围授权。

## 验收标准

1. 数据中心导航出现“用户洞察”，并可在店铺视角和产品/SKU 视角之间切换。
2. 页面先选平台，再在“用户分析 / 竞品分析 / 规则与建议”三个页内 Tab 间切换；选择不被无关页面重置。
3. 用户分析能按平台展示用户、商品、视频和直播市场事实，并说明每个维度的来源、范围、覆盖率和更新时间。
4. 系统能推荐平台类目，但未经产品或运营人员确认时不会进入自动采集计划。
5. 工作日采集只访问已登记页面；周末不自动运行，人工重试仍需权限并进入审计记录。
6. 采集器和表单不存在密码、Cookie、Token、验证码或浏览器历史字段；D1 中不存在完整页面内容和会话数据。
7. 竞品分析区分人工核心竞品和系统候选竞品；候选展示命中规则，未经确认不进入核心对比。
8. 产品和运营能分别维护本 App 的版本化规则；复制另一个 App 的规则不会修改源规则。
9. 同一指标在不同平台不可比时采用平台分列和说明，不输出伪合计；共同口径可在跨平台摘要中对照。
10. 每条建议显示证据、规则版本、覆盖率、置信度、新鲜度、限制和“仅供参考”，且没有自动执行按钮。
11. 登录失效、结构变化、部分缺失、过期、失败和不支持状态均保留最后成功快照或清晰空状态，不把缺失当作 0。
12. API 对查询、规则维护、候选确认和设备采集分别校验组织、App、范围、版本和令牌权限。
13. CI 使用固定页面、导出和响应夹具验证解析、标准化、幂等、缺失字段和结构变化，不登录真实店铺。
14. 生产验收在已授权、已登录的真实平台页面单独执行，记录平台、类目、页面结构版本和结果，不进入 CI。
15. 键盘、焦点、加载、空、错误、禁用、只读、390–1440px、WCAG AA 和钉钉 WebView 均通过验收。

## 上线与回滚

- 导航、查询、规则和采集入口使用独立功能开关，先在本地和预览环境启用。
- 新增独立 D1 表和向前兼容迁移，不覆盖现有销售事实、产品主档或运营方案。
- 环境能力清单、集成注册表、生成模块、API 契约、迁移和回滚说明必须与代码同批更新。
- 各已登记且具备市场页面的平台分别通过适配器端到端验证后启用；功能开关允许逐平台灰度，但正式交付不得把某个平台的数据冒充为跨平台结果。
- 回滚时关闭入口和采集计划，保留规则、审计和最后成功快照供恢复，不删除其他 App 数据。
- 未经单独确认不部署生产、不创建真实平台动作、不写入凭证，也不更改远程仓库设置。
