# 老板行动台与跨 App 部门协同功能设计书

## 用户任务

老板进入公司首页后，第一眼要知道今天需要自己拍板或协调的事项，而不是先解释一组汇总数字。部门员工进入协同工作台后，要能判断哪些事项需要本部门接收、本人推进、等待其他部门或等待自己验收。

业务 App 用户在发现交接、风险、决策或数据问题时，应从当前业务记录直接发起协同，并保留返回来源的路径。协同工作台不替代业务 App 详情。

## 信息层级

### 老板行动台主信息

1. 今天需要拍板的事项、截止时间、推荐方案和不决策的影响。
2. 已阻塞或逾期的跨部门事项、唯一主责部门、主负责人和需要总经办协调的动作。
3. 会影响战略关键结果或重点项目的高影响风险。
4. 上次查看后新出现、升级、退回、重新打开或解除的重要变化。

### 部门协同主信息

1. 当前状态和下一步由谁完成。
2. 发起部门、主责部门、主负责人和协同部门。
3. 截止时间、业务影响、来源 App 和完成标准。
4. 接收、执行、阻塞、提交验收和验收历史。

### 辅助信息

- 类型、影响等级、来源 App、部门、负责人、状态和截止时间筛选。
- 战略关键结果、重点项目和相关历史事项关联。
- 钉钉待办同步状态和最近同步时间。

### 低频信息

- 幂等来源键、版本、内部 ID、归档信息和完整活动字段。
- App 接入健康、同步延迟和协议版本。

## 页面结构

### 导航

- 总经办导航在“公司经营”分组保留“公司首页”，并新增“部门协同”，位置在“经营检查”之后、“业务 Apps”之前。
- 首次生产发布仅对总经办自动显示“部门协同”；普通员工继续看到原产品全周期外壳。后续可通过 `executiveCollaborationHub` 功能开关逐步开放部门入口。
- 普通员工沿用原产品全周期外壳，只新增授权可见的“部门协同”；不会看到战略中心、重点项目、部门激励、经营检查或业务 Apps。
- 各业务 App 继续使用自己的左侧分组和页面；不增加第二套常驻顶部二级导航。

### 公司首页：老板行动台

页头标题为“公司首页”，说明改为“处理今天需要拍板、协调和升级的经营事项”。不使用大号英雄数字或六等分指标条作为首屏主入口。

首屏使用稳定的双栏工作区：

- 左侧约三分之二宽度为“今天需要处理”，依次显示待决策和需要协调的阻塞事项。
- 右侧约三分之一宽度为“重要变化”，展示最近 7 天进入、升级、解除和重新打开的高影响事项。
- 每行使用固定信息顺序：行动原因、标题、影响或建议、主责和截止、来源标签、明确动作。
- 没有事项时显示“当前没有需要总经办处理的升级事项”，并保留进入全量协同和战略中心的入口。

首屏下方保留战略执行区，但改为“战略结果与执行缺口”：

- 每条战略展示必达结果核验、异常部门承诺、关联协同阻塞和最近事实时间。
- 点击战略或关键结果进入战略中心对应下钻。
- 不使用里程碑数量推导业务结果；来源 App 事实只能作为证据和预警。

再下方保留重点项目、激励和月报的变化摘要。内容默认折叠为高密度行，用户主动进入对应 App 查看全量信息。

### 部门协同工作台

页头包含标题、作用说明、“发起协同”主操作和最后更新时间。页面使用一个筛选工具栏和一个主列表，不使用同尺寸卡片瀑布流。

状态视图使用可键盘切换的标签组：

- 待我部门接收
- 执行中
- 等待其他部门
- 待我验收
- 我参与的
- 已完成

工具栏提供关键词、来源 App、类型、影响等级、部门和到期范围筛选。普通员工不显示全公司部门筛选；总经办可以切换“我的范围 / 全公司”。

主列表使用 `DataTable`，最低宽度 1120 像素，列为：事项、来源、发起/主责、主负责人、状态、影响、截止、下一步、操作。中文表头和日期保持单行，窄屏水平滚动。

选中行后在 ≥1180 像素显示右侧详情面板；面板不会覆盖列表，关闭后焦点回到原行。在更窄宽度使用全屏详情层，避免表格旁的狭窄详情。详情内容顺序为：当前行动、责任关系、业务影响、完成标准、来源与关联、活动时间线、钉钉同步。

### App 内发起入口

业务 App 的异常行、任务行或数据问题行增加统一的次要操作“发起协同”。点击后打开轻量表单，预填来源、类型、标题、业务影响和证据；用户必须确认主责部门、主负责人、截止时间和下一步。

App 适配器仅构造草稿。保存成功后显示事项 ID、当前接收状态和“查看协同详情”，不在 App 内复制活动时间线或状态流转按钮。

首批入口：

- 产品进度：跨部门任务、逾期关键任务、阶段准入阻塞。
- 供应链：异常库存、采购/付款待映射、质量整改。
- 数据中心：数据过期、待映射、对平异常、数据源登录或结构异常。
- 品牌内容：只完成适配器契约和领域测试；业务 UI 上线后接入行操作。

### 业务 Apps 页面

“业务 Apps”从纯目录改为接入健康表。每个 App 行展示协议状态、数据新鲜度、未关闭协同、高影响阻塞和最近上报时间。点击主按钮进入 App，点击协同数量进入已按 App 过滤的部门协同工作台。

未实现协同适配器的 App 显示“业务已登记 · 协同待接入”，不能伪装为已连接。

## 交互流程

### 发起协同

1. 用户从 App 行操作或协同工作台点击发起。
2. 表单展示只读来源和可编辑标题、说明、下一步、业务影响、影响等级、主责部门、主负责人、协同部门、截止时间及可选战略关联。
3. 人员和部门使用 `OrgSelect`，不允许自由文本创建虚拟人员。
4. 保存前验证主责、下一步、影响和截止；协同部门自动排除主责部门。
5. 命中未关闭幂等事项时，不创建重复记录，提示“该来源已有进行中的协同事项”并进入详情。
6. 保存成功后进入 `pending_acceptance`，焦点移动到成功提示中的“查看事项”。

### 接收与退回

1. 主责部门成员打开待接收事项。
2. “确认接收”要求选择唯一主负责人；如果当前已有有效主负责人，可以直接确认。
3. “退回补充”要求填写缺失信息或责任边界，确认后状态变为 `returned`。
4. 发起方修改后执行“重新提交”；历史退回原因始终可见。

### 执行、阻塞与验收

1. 主负责人在详情首部看到当前允许动作。
2. 标记阻塞时填写阻塞原因、需要谁协调和预计恢复时间；高影响阻塞立即进入老板行动台。
3. 恢复执行要求填写解除说明。
4. 提交验收要求填写完成结果；可附来源 App 中的结果链接，不在协同平台上传大文件。
5. 发起方验收关闭，或填写原因重新打开。

### 决策事项

1. 决策事项必须显示问题、业务影响、推荐方案、备选方案和截止时间。
2. 指定决策人可以执行“同意推荐方案”“选择备选方案”或“退回补充”；选择结果和说明进入活动记录。
3. 决策关闭后，来源重点项目或协同事项可以根据明确操作继续推进；系统不自动修改业务状态。

### 归档

1. 发起方或总经办在更多操作中选择“归档事项”。
2. 归档确认说明影响并要求原因；已接收事项不能物理删除。
3. 归档后从默认列表移除，详情保持只读，可由总经办或原发起方查看。

### 钉钉待办

1. 事项接收并确定主负责人后显示“同步到钉钉待办”。
2. 用户主动执行同步；平台调用现有钉钉待办服务，使用稳定来源 ID。
3. 后续标题、截止时间和关闭状态更新同一待办。
4. 钉钉失败不撤销平台状态；详情显示失败原因和“重试同步”。
5. 读取钉钉完成状态时，只回流到协同事项的提交/关闭动作边界，不修改来源 App 业务状态。

## 组件复用

- `PageHeader`：页面标题、说明、全局更新时间和主操作。
- `DataTable`、`TableActions`：协同列表、App 健康表和动作菜单。
- `Button`：主操作、普通操作、禁用原因和加载状态。
- `OrgSelect`：主责部门、主负责人、协同部门和筛选。
- `DatePickerField`：截止时间和预计恢复时间。
- `ConfirmDialog`：取消、归档、重新分配和其他高影响动作。
- `HealthBadge`：只用于已有战略/项目健康语义；协同状态使用独立文字状态标识。
- `Modal`：只用于新增/编辑表单和窄屏详情；桌面详情优先侧面板或展开区。

## 新增组件

### CollaborationStatusBadge

- 输入：`status`、可选 `compact`。
- 输出：中文状态、语义图标和统一色调。
- 覆盖未知、加载和归档状态；不能只靠颜色区分。
- 保留在 `src/features/collaboration/`，除非第二个非协同消费者需要相同状态语义。

### CollaborationFilters

- 输入：当前视图、筛选值、授权范围和 App 注册表。
- 输出：结构化过滤条件；不自己请求数据。
- 所有控件有可见标签，清除筛选后焦点返回触发按钮。

### CollaborationTable

- 输入：事项、选中 ID、当前用户、允许动作和事件回调。
- 使用共享 `DataTable`，保持单行表头、稳定列宽、键盘打开详情和空状态。
- 不在单元格中嵌套卡片或活动时间线。

### CollaborationDetailPanel

- 输入：事项、活动记录、允许动作、加载和错误状态。
- 桌面为旁侧详情，窄屏为全屏层；关闭后恢复原触发元素焦点。
- 顶部只展示当前行动和责任，低频 ID 与版本放到“记录信息”。

### CollaborationEditor

- 输入：可选来源草稿、现有事项、组织缓存、战略/项目选项和保存回调。
- 新建和编辑共享字段校验；来源身份只读。
- 保存中禁用重复提交；版本冲突保留用户输入并提供刷新比较。

### ExecutiveActionList

- 输入：行动投影和导航回调。
- 按行动原因分组为“待拍板”“需要协调”“高影响逾期”，每行只提供一个主要动作。
- 空数据时展示清晰完成状态，不显示装饰性数字。

### AppCollaborationButton

- 输入：App 适配器输出的标准草稿、禁用原因和保存成功回调。
- 业务中立，不知道产品、库存、数据源或内容字段。
- 只有产品、供应链和数据中心三个真实调用方接入后，才评估移动到 `src/ui`；首版保留在协同功能目录。

## 页面状态

- 加载：保留筛选栏、表头和详情列宽的骨架；老板行动台的不同队列可以独立完成加载。
- 空数据：区分“当前无需处理”“当前筛选无结果”“尚未创建协同事项”和“没有权限查看全公司”，提供对应下一步。
- 错误：说明失败范围、request ID、是否仍显示缓存和重试动作；不能用整页空白代替错误。
- 无权限：显示当前人员和部门范围、允许查看的事项类型以及权限来源；服务端返回相同边界。
- 禁用：在按钮附近展示“等待接收部门确认”“只有主负责人可以提交验收”等具体原因。
- 成功：展示新状态、下一责任人和下一步，不只显示“保存成功”。
- 冲突：保留本地输入，显示服务器最新更新时间和操作者，用户刷新比较后重新提交。
- 来源不可用：协同详情继续可用，来源按钮禁用并说明 App 尚未上线或当前无权限。
- 钉钉失败：平台状态保持成功，钉钉区域显示安全错误摘要、失败时间和重试按钮。

## 响应式与钉钉 WebView

- ≥1280px：固定左侧导航；老板行动台双栏；协同列表与详情左右分栏。
- 1024–1279px：老板行动台主次区上下排列；协同列表全宽，详情使用覆盖主区的全高层。
- 760–1023px：侧栏折叠；筛选折叠为两行；表格水平滚动且操作列保持可用。
- <760px：单列；状态视图横向滚动；事项使用高密度行列表替代表格；详情全屏；底部操作区适配安全区。
- 使用 `100dvh`、`env(safe-area-inset-*)` 和唯一主滚动容器，避免钉钉 WebView 双滚动。
- 不使用大请求 `keepalive`；列表分页和详情按需加载。

## 交互文案

- 页面说明：“跨 App 接住责任、推进执行，并保留可验收的部门协同记录。”
- 老板行动台说明：“处理今天需要拍板、协调和升级的经营事项。”
- 待接收空状态：“当前没有需要你部门确认接收的事项。”
- 老板空状态：“当前没有需要总经办处理的升级事项。”
- 重复来源：“该业务记录已有进行中的协同事项，已为你打开。”
- 版本冲突：“事项已被其他同事更新，你的输入尚未丢失。请比较最新内容后重新提交。”
- D1 不可用：“共享协同数据暂不可写入，当前显示上次缓存。请稍后重试。”
- 来源不可用：“来源 App 当前不可进入，协同记录仍可继续处理。”
- 阻塞按钮：“标记阻塞并请求协调”。
- 提交按钮：“提交发起方验收”。
- 归档确认：“归档后将从默认列表隐藏，但不会删除历史责任和活动记录。”

## 无障碍

- 页面焦点顺序为页头、状态视图、筛选、列表、详情和动作；跳过链接继续指向主要内容。
- 状态视图使用 `tablist`、`tab` 和 `tabpanel` 语义，方向键可以切换。
- 表格行详情按钮包含事项标题；状态、影响和逾期不只靠颜色表达。
- 打开和关闭详情、编辑器、确认层时正确转移并恢复焦点。
- 表单错误与字段通过 `aria-describedby` 关联；保存错误使用 `role="alert"`，成功使用 `role="status"`。
- 正文和辅助文字达到 WCAG AA；焦点环不被表格滚动容器裁切。
- 状态变化使用 150–200ms 过渡，并为 `prefers-reduced-motion` 提供即时切换。

## 视觉验收

1. 1440×900：老板行动台待决策、阻塞、重要变化和战略结果区；部门协同列表与详情分栏。
2. 1280×800：长中文标题、部门名称、截止日期和行操作无错位。
3. 1024×768：双栏正确降为单栏，详情层不被侧栏或顶部栏裁切。
4. 390×844：标签、筛选、列表、全屏详情和底部动作无双滚动并适配安全区。
5. 数据状态：加载、首次空数据、筛选空数据、错误、403、409、D1 501、来源不可用和钉钉失败。
6. 权限状态：总经办全公司、普通主责部门、协同部门、无关部门和只读会话。
7. 键盘：跳过导航、切换视图、筛选、打开详情、确认接收、填写阻塞、提交验收和关闭详情。
8. 对比度：正文、辅助文字、禁用文案、状态标识、焦点环和按钮满足 WCAG AA。
