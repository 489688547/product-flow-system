# 品牌内容协同规格阶段任务

## 执行规则

- 规格复核通过前不写业务行为代码。
- 任何范围、权限、数据口径或集成边界变化先更新 PRD 与设计书。
- 正式开发任务必须先写失败测试，再做最小实现和聚焦验证。
- 当前任务只提交品牌内容协同规格与绿联 NAS 计划中登记。

## 任务

- [x] 调研现有平台与历史素材分析问题
  - 依赖：无。
  - 文件：`PRODUCT.md`、`DESIGN.md`、`docs/product/`、`docs/platform/`、现有 App 壳与测试。
  - 输入：用户提供的七月素材分析任务、数据中心任务、平台截图和绿联 NAS 调研结果。
  - 输出：首版范围、平台边界、角色与风险口径。
  - 失败测试：不适用；本项为只读调研。
  - 实现步骤：核对仓库规则、历史口径错误、数据中心供数边界和绿联官方能力。
  - 验证：首版选择“内容经营闭环型”，抖音优先，快手/小红书标准导入，NAS 使用只读索引。
  - 提交：与本规格提交合并。

- [x] 确认视觉结构
  - 依赖：调研完成。
  - 文件：可视化 brainstorming 草图，不进入产品代码。
  - 输入：经营执行平台左侧导航和用户视觉反馈。
  - 输出：品牌内容协同九个左侧页面，删除顶部重复导航。
  - 失败测试：不适用；本项为交互评审。
  - 实现步骤：展示结构草图，收集反馈，生成删除顶部导航后的修正版。
  - 验证：用户确认“其他没问题可以开发”。
  - 提交：草图保留在本地 brainstorming 目录，不纳入 Git。

- [x] 编写并自检书面规格
  - 依赖：视觉结构确认。
  - 文件：`prd.md`、`design.md`、`plan.md`、`tasks.md`、`docs/platform/integration-registry.json`。
  - 输入：已确认范围与仓库模板。
  - 输出：可复核、无未完成标记的完整功能规格。
  - 失败测试：`npm run check:governance` 在缺少 `plan.md` 和 `tasks.md` 时失败。
  - 实现步骤：补齐四份文件，扫描未完成标记、矛盾、歧义和遗漏，登记绿联 NAS。
  - 验证：`npm run check:governance`、`npm run check:integrations`、`git diff --check`。
  - 提交：`docs: design brand content collaboration app`。

- [ ] 业务负责人复核书面规格
  - 依赖：规格提交完成。
  - 文件：`prd.md`、`design.md`。
  - 输入：本次提交的可点击文件。
  - 输出：明确批准，或逐项修改意见。
  - 失败测试：规格未获明确批准时，正式实施计划不得开始。
  - 实现步骤：业务负责人确认页面、角色、指标、数据中心与 NAS 边界。
  - 验证：收到明确的书面批准。
  - 提交：如有修改，使用独立规格修订提交。

- [ ] 编写正式实施计划
  - 依赖：业务负责人明确批准书面规格。
  - 文件：本目录 `plan.md`、`tasks.md` 以及需要更新的产品、平台、API 和架构文档。
  - 输入：已批准 PRD 与设计书。
  - 输出：精确文件职责、接口、迁移、回滚、失败测试和逐任务提交计划。
  - 失败测试：计划缺少任何验收标准映射、错误状态或接口字段时不得执行。
  - 实现步骤：使用 writing-plans 流程覆盖所有规格要求，并完成类型与占位符自检。
  - 验证：规格中的每项验收标准都有对应实现任务和验证命令。
  - 提交：`docs: plan brand content collaboration app`。
