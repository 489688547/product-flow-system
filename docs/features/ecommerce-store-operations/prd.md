# 电商店铺运营 App PRD

## 文档状态

- 状态：待书面复核
- 负责人：运营主管 / 运营部
- 最近更新：2026-07-18

## 背景与问题

运营主管每月会确定重点产品，但现有重点产品、店铺打法、执行动作、跨部门支持和复盘分散在聊天、表格及平台后台。运营能够看到结果，却难以持续说明“当前事实是什么、目标是什么、最重要的问题是什么、为什么这些对策能解决问题、用什么数据验证”。主管也无法在同一处判断方案质量、执行卡点、团队负荷和辅导重点。

公司已经建设产品全周期、供应链管理和数据中心，另有品牌内容协同边界。新 App 应编排这些能力，而不是复制产品主档、库存、素材或经营事实。

## 目标

1. 新增与现有业务 App 同级的“电商店铺运营”App。
2. 以“现状证据 → 目标 → 核心问题 → 对策 → 检测指标 → 执行 → 复盘”管理重点产品经营闭环。
3. 支持一个重点产品在同一月份拆分到多个平台、店铺和运营负责人，同时保留产品级汇总。
4. 由 AI 检查方案完整度和因果链，向运营提供可执行的优化建议；运营主管保留最终批准权。
5. 从数据中心读取销售、退款、毛利、投放及数据质量信息，形成可追溯的方案证据和复盘结果。
6. 让运营发起跨部门请求，目标部门正式接受或退回并说明原因，状态持续回流。
7. 为运营主管提供责任分配、工作负荷、方案质量、复盘及时性、协同关闭情况和辅导记录。
8. 向独立绩效管理 App 提供经过权限校验的任务、结果和复盘证据引用，不在本 App 保存绩效分数。

## 非目标

- 不在本 App 直接抓取快麦、抖音、淘宝、拼多多、快手、小红书或广告平台页面。
- 不复制数据中心日事实、产品全周期主档、供应链库存或品牌内容主档。
- 不自动调整价格、预算、广告计划、库存、内容排期或平台商品设置。
- AI 不批准方案、不评价员工绩效、不替代主管做放量或止损决策。
- “团队管理”不包含考勤、薪酬、招聘、劳动关系、员工自评、绩效评分或单一 GMV/ROI 排名；这些数据由独立绩效管理 App 负责。
- 一期不把聊天、会议或自由文档作为唯一经营事实来源。

## 用户与权限

### 运营主管

查看运营范围内全部店铺和重点产品；制定月度重点产品及总目标；分配平台、店铺和运营负责人；批准或退回方案；处理资源冲突；关闭月度经营周期；维护团队责任与辅导记录。

### 平台或店铺运营

查看并编辑自己负责的经营范围；维护现状、目标建议、核心问题、对策、检测指标、执行记录和周复盘；处理 AI 建议；发起跨部门协同。不能修改他人的方案、数据中心事实或主管最终结论。

### 协同部门负责人

查看发给本部门的请求；接受或退回并填写原因；更新承诺时间和交付状态；关联所属 App 的实体。不能修改运营方案和经营事实。

### 总经办及相关部门

总经办可查看全局经营闭环和重大偏差；产品、品牌、供应链、客服和财务按组织与业务范围查看相关内容。财务事实和团队辅导记录遵循最小权限，前端不可见不代替服务端授权。

## 当前流程

1. 主管在聊天或表格指定重点产品。
2. 运营分别写活动、投放或内容动作，现状、问题和目标常被混在一起。
3. 跨部门支持通过聊天推进，接受、退回和交付缺少稳定记录。
4. 运营从多个后台抄数复盘，口径和截止时间不一致。
5. 主管主要根据结果追问，难以区分方案问题、执行问题、协同阻塞或数据缺失。

## 目标流程

1. 主管创建月度重点产品经营周期，选择产品、总目标、平台店铺范围和负责人。
2. 系统从数据中心读取截止昨天的经营事实，并显示来源、覆盖率和新鲜度。
3. 每名运营在负责范围内完成现状证据、目标、1–3 个核心问题、对策和检测指标。
4. AI 对当前版本给出缺口、风险和优化建议；运营修改后可再次点评。
5. 主管批准方案或带原因退回；批准后的版本进入执行，后续修改形成新版本。
6. 对策需要其他部门支持时发起协同请求；目标部门接受或退回，交付状态与目标 App 链接持续回流。
7. 数据中心按日更新检测指标；系统提示异常，不要求运营每天重复填报数据。
8. 运营每周提交证据、结果、判断和下一步；AI 对照方案与实际数据提出复盘建议。
9. 月末由主管决定继续放量、调整打法、转日常经营或停止，并记录团队辅导和可复用方法。

## 业务规则

1. 主导航中“电商店铺运营”位于数据中心之后、平台之前；App ID 为 `ecommerce-operations`。
2. 左侧页面为经营驾驶舱、重点产品经营、跨部门协同、团队管理和经营方法库。
3. 一个 `OperatingCycle` 表示一个产品在一个自然月的经营周期；同一产品和月份只有一个有效主周期。
4. 主周期包含一个或多个 `StorePlan`。每个子方案绑定平台、店铺、主运营和目标；产品级目标由主管汇总确认。
5. 重点产品、方案、执行和复盘属于同一工作台，不建立相互割裂的平级页面。
6. 现状必须包含数据证据或明确的非数据事实；数据证据记录来源、时间范围、截止时间和质量状态。
7. 每个子方案保留 1–3 个当前核心问题；问题必须关联证据并解释其与目标的因果关系。
8. 每个对策必须绑定核心问题、负责人、适用范围、计划时间、资源、风险、止损线和至少一个检测指标。
9. 检测指标区分过程指标与结果指标；缺失值使用 `null`/“无数据”，不得使用 0 伪装完整。
10. AI 点评检查证据、目标、问题、对策、指标、风险和协同完整度，只向运营提供建议，不改变状态或代替主管批准。
11. 主管可带原因退回或覆盖 AI 的完整度警告；覆盖原因进入审计记录。
12. 销售按订单创建时间、`Asia/Shanghai`、默认截止昨天；常规经营分析排除“其它”、空平台和未知平台。
13. 数据未对平、覆盖率不足或已过期时，AI 和页面只能给出条件性判断，不得输出确定归因。
14. 协同状态为待接受、已接受、已退回、执行中、已交付、已关闭；退回、改期和关闭必须说明原因。
15. 周复盘不覆盖历史方案或结果；方案、AI 点评、执行记录和复盘均保留版本与时间。
16. 团队管理按组织身份展示责任范围、活跃周期、超期动作、方案质量缺口、复盘及时性、协同关闭和辅导事项，不产生或保存绩效分数。
17. 绩效管理 App 只能通过稳定证据引用读取已授权任务、结果和复盘摘要；引用包含来源 App、实体、版本、验收状态和取得时间，不允许绩效 App 回写经营任务。

## 数据定义

- `OperatingCycle`：产品、月份、主管、产品级目标、总体状态和月度决策。
- `StorePlan`：平台、店铺、主运营、方案版本、审批状态和负责范围。
- `EvidenceSnapshot`：数据中心或业务事实的引用、值、范围、截止时间、覆盖率和取得时间；用于保存当时判断依据，不替代源事实。
- `Goal`：指标编码、基准值、目标值、方向、截止时间和适用范围。
- `KeyIssue`：问题陈述、优先级、证据引用、因果解释、状态和适用范围。
- `Countermeasure`：问题引用、动作、负责人、时间、资源、风险、止损线和协同请求。
- `Monitor`：过程/结果类型、指标编码、来源、检测频率、预期变化和阈值。
- `ExecutionUpdate`：对策进展、实际值、证据、偏差、阻塞和下一步。
- `Review`：数据窗口、结果、问题是否解决、对策是否有效、结论和下一轮决定。
- `AiReview`：输入版本、各维度判断、建议、风险、数据限制、模型/规则版本和创建时间。
- `CollaborationRequest`：来源方案、目标部门、目标 App、承诺、接受/退回原因、目标实体引用和状态。
- `TeamResponsibility`：组织成员、平台、店铺、重点产品和生效时间。
- `CoachingRecord`：主管、成员、观察事实、改进行动、复核日期和可见范围。
- `PerformanceEvidenceRef`：提供给绩效管理 App 的来源 App、实体类型、实体 ID、版本、负责人、验收状态、摘要和取得时间。

## 异常与边界

- 数据中心不可用：保留上次快照并显示截止时间，允许编辑方案，暂停生成新的确定性数据结论。
- 数据缺失或过期：显示受影响指标和范围，禁用“数据已验证”类结论。
- AI 服务未配置或失败：保存方案不受影响，显示重试入口和人工检查清单，不伪造 AI 点评。
- 产品或店铺未映射：周期可保存为草稿，但不能提交主管批准。
- 重复周期或并发编辑：服务端使用版本号拒绝覆盖，要求刷新后合并。
- 协同目标 App 未接入：允许目标部门接受/退回并填写承诺，目标实体显示“尚未关联”，不伪造已创建任务。
- 权限不足：页面解释只读原因，服务端返回 403；组织身份变化后重新计算权限。
- 跨月执行：原月周期保留，未完成对策必须由主管明确结转到下月，不能静默移动。

## 验收标准

1. 用户可从数据中心之后进入“电商店铺运营”，看到五个入口且不存在独立“打品方案/执行/复盘”平级页面。
2. 主管可为同一重点产品创建一个月度周期，并拆分至少两个平台店铺子方案和不同负责人。
3. 运营只能编辑自己负责的子方案；主管可批准或带原因退回；服务端拒绝越权写入。
4. 子方案缺少现状证据、目标、核心问题、对策或检测指标时不能提交批准，并获得明确缺口说明。
5. AI 点评返回结构化建议，运营修改后形成新版本；AI 不改变审批状态。
6. 数据中心数据标明创建时间口径、截止日期、覆盖率和质量状态；“其它”不进入正常经营判断。
7. 一个对策可发起跨部门请求；目标部门能接受或退回并说明原因，运营和主管看到最新状态。
8. 周复盘能对照原目标、核心问题、对策和指标，生成继续、调整、加码或停止建议，历史版本不被覆盖。
9. 团队管理展示责任范围、负荷和过程质量，不包含考勤薪酬或自动绩效排名。
10. 数据中心或 AI 不可用时页面仍可编辑和查看历史记录，且不会把缺失数据当成 0。
11. 键盘、焦点、空/错/禁用/只读/过期状态、390–1440px 和钉钉 WebView 均通过验收。
12. 绩效管理 App 可引用运营任务和复盘证据，但电商运营 API 不返回员工自评、建议分或主管评分字段。

## 上线与回滚

- App 注册、导航和 API 使用独立功能开关；默认先在本地和预览环境启用。
- 新增独立 D1 表，不迁移、不覆盖数据中心、产品、供应链或品牌数据。
- AI 适配器无凭证时保持禁用；凭证只存在服务端环境变量。
- 监控 API 错误、版本冲突、数据新鲜度、AI 失败和协同超期。
- 回滚时关闭功能开关并停止 AI 请求；保留经营周期和审计数据供恢复，不删除其他 App 数据。
- 未经单独确认不部署、不配置生产凭证、不发送真实钉钉通知。
