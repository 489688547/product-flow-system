# 商品主数据销量视图设计

## 设计结论

扩展现有共享商品目录，而不是新增“商品销量”App。用户仍在商品主数据完成搜索、筛选、同步和质量判断，销售事实成为每个商品的经营上下文。

## 信息架构

页面由三层组成，不增加嵌套卡片：

1. 经营状态带：商品数、筛选范围销量、销售额、销售覆盖商品数和两个来源更新时间。
2. 查询工具栏：商品搜索、平台、日期预设、自定义起止日期、导入与同步动作。
3. 商品表：商品、SKU/69 码、分类/品牌、销量、ERP 成本、业务关联。

“ERP 状态”从主表移除。启停与来源缺失仍保留在数据层，后续可在数据质量视图处理，不占老板日常经营扫描的核心列。

## 视觉层级

- 延续浅色、克制、高密度工作台，使用现有系统字体、蓝色主操作和 Lucide 图标。
- 状态带使用一条连续容器和细分隔线，不使用独立 KPI 卡片或装饰阴影。
- 销量列为表格第一数字列：主值 15px/700，销售额和平台构成为 11–12px 辅助信息。
- 数量与金额右对齐并启用 tabular numbers；零销量采用中性色，不能看起来像错误。
- 来源更新时间靠右显示“商品档案 / 销售数据至 / 入库时间”，避免把两条同步链路或入库时间与数据覆盖日期混为一谈。

## 筛选交互

- 平台：单选，默认“全部平台”；选项来自服务端 `meta.sales.availablePlatforms`。
- 日期：最近 7 天、最近 30 天、本月、上月、自定义。默认最近 30 天。
- 自定义：只有选择“自定义”后显示两个原生日期输入；起止日期任一变更即校验，合法后查询。
- 搜索、分类和关联状态继续在当前已加载目录内即时过滤；平台和日期必须服务端查询。
- 网络查询使用延迟值避免搜索影响，但日期/平台选择立即生效；请求竞态由客户端忽略旧响应。

## 表格行为

- 商品名称最多两行；编码、平台、日期和数字不换行。
- 销量列显示 `quantity`，辅助显示净销售额；多个平台时显示前两个平台及占比/数量概览。
- 成本列保持现有权限裁剪；无成本权限时 API 不返回成本字段。
- 业务关联继续显示产品全周期关联和供应商覆盖。
- 表格默认 50 条分页；筛选条件变化回到第一页。
- 默认按销量、净销售额降序排列，同值再按商品名稳定排序；搜索、分类和关联筛选后保持同一经营优先级。

## 页面状态

- 初次加载：状态带与表格骨架。
- 销量刷新：保留商品目录和上次销量，工具栏显示“销量更新中”。
- 销量错误：保留上次成功数据，在工具栏下方提示失败和重试；目录不清空。
- 无目录：引导同步快麦商品或导入文件。
- 无筛选结果：保留筛选条件并提供清除入口。
- 无销量：商品仍显示，销量为 0，状态带显示当前范围暂无销售。
- 覆盖不足：显示未匹配销售行数，提示先核对 69 码，不把事实强行归属商品。
- 覆盖滞后：查询结束日晚于范围内最新销售事实日期时，显示“销售事实仅更新至 YYYY-MM-DD”，不宣称截止昨天。
- 无写权限：筛选和读取可用，导入隐藏，同步按钮禁用并说明原因。

## 响应式与无障碍

- 1180px 以上完整显示；900px 下状态带可横向滚动；640px 下工具栏分层；390px 下日期输入纵向排列。
- 表格最小宽度由列内容决定，只在表格容器内横向滚动。
- 所有筛选有可见 label / accessible name；日期错误通过 `role=alert` 宣布。
- 控件保留 focus-visible；状态不依赖颜色；按钮禁用提供原因。

## 共享能力评审

结论：扩展现有能力。

- `/api/platform/v1/product-catalog` 已被数据中心、产品全周期和供应链消费，销量聚合属于同一共享商品读模型。
- `product_sales_daily` 已是销售唯一事实表，新增复制表会造成口径和回填漂移。
- UI 只在商品主数据使用，保留为数据中心 feature；日期范围纯函数放在领域层，API 查询放在 state 层。
