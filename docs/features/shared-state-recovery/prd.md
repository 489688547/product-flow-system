# 产品全周期共享状态恢复 PRD

## 背景与问题

产品全周期页面启动时会优先读取浏览器缓存；当缓存带有 `productFlowStateDirty=1` 时，旧实现会在读取 D1 之前把整份本地状态写入 `/api/state`。普通共享状态写入没有基线版本、写前快照或审计，因此旧分支、旧标签页和默认演示数据可以覆盖较新的公司状态。

## 目标

- 恢复事故前最后一份有效产品全周期状态，不回滚 D1 中其他业务表。
- 任何客户端必须先读取服务器版本，才能保存整份共享状态。
- 启动时的本地脏缓存只能保留为恢复副本，不能自动写入公司数据。
- 每次共享状态写入都生成写前快照和审计记录，可定位、可回滚。
- 旧分支和旧标签页因为不携带基线版本而被服务端拒绝，不能继续覆盖生产数据。
- 同一服务器基线只能被一个写入原子推进，其他并发写入必须失败。
- 规范化业务状态没有变化时不发送保存请求，不生成无意义快照和审计。

## 非目标

- 不改造产品全周期的业务字段和页面布局。
- 不触发钉钉、快麦或其他外部平台动作。
- 不恢复已被业务人员明确删除且不在事故前最后有效状态中的记录。
- 不新建第二套生产数据库或绕过现有生产数据网关。

## 用户与权限

- 已登录员工可以读取共享状态。
- 只读账号不能写入。
- 普通业务写入使用当前公司会话，并接受版本、快照和审计约束。
- 事故恢复继续使用最高权限个人令牌、15 分钟解锁、版本检查、写前快照和审计的生产数据网关。

## 当前流程

1. 页面从 `localStorage` 初始化，缺少缓存时使用演示默认值。
2. 正式站点如果发现本地脏标记，先 POST 整份本地状态。
3. `/api/state` 不检查服务器基线，直接删除并重建全部状态分片。
4. 页面加载后组织同步、待办刷新或任意本地变化继续触发整份自动保存。

## 目标流程

1. 页面可以读取本地缓存用于首屏，但共享写入保持锁定。
2. 页面先 GET `/api/state`，保存返回的 `updatedAt` 作为唯一写入基线。
3. 若本机存在未同步缓存，先保存为本机恢复副本，再以服务器状态为准；不自动上传。
4. 用户产生新修改后，POST 同时提交 `baseUpdatedAt`。
5. 服务端在同一个 D1 原子批次中比较并推进版本、替换状态分片；缺少、不一致或已被并发推进时返回 409，任何状态分片都不写入。
6. 版本一致时先保存写前快照和待完成审计，再写状态，最后完成审计并返回新的 `updatedAt`。
7. 事故恢复通过现有生产数据网关写入已提取的最后有效状态。

## 业务规则

- D1 公司共享状态是在线运行时的唯一事实源。
- 本地缓存不是服务器快照，不能作为自动恢复依据。
- 未成功读取服务器状态时禁止自动保存默认数据、本地缓存、组织同步结果或待办刷新结果。
- `/api/state` 的每次 POST 必须携带最近一次 GET/POST 成功返回的 `baseUpdatedAt`。
- 版本冲突必须显式失败，客户端不得静默重试或覆盖。
- 客户端用规范化业务状态指纹判断是否需要保存；`orgCache.syncedAt`、`orgCache.expiresAt` 等易变缓存时间不构成业务修改。
- 服务端版本比较、版本推进和全部分片替换必须属于同一个原子事务，不能采用“先读检查、后无条件写入”。
- 每次成功写入前必须生成完整状态快照；审计记录操作者、原因、前后版本、请求 ID、结果和时间，不记录凭据。
- 数据恢复只替换产品全周期共享状态，不回滚商品目录、组织、会话、销售、供应链或平台连接表。

## 数据定义

- `baseUpdatedAt`：客户端最后确认的服务器状态版本，ISO 8601 时间字符串。
- `updatedAt`：服务端成功写入后生成的新版本。
- `product_flow_state_parts`：当前公司共享状态分片。
- `production_data_snapshots` / `production_data_snapshot_parts`：每次写入前的完整可回滚状态。
- `production_data_audit`：普通共享写入和运维修复的审计记录。
- 本机恢复副本：仅保存在浏览器本地，不参与自动同步，也不得成为共享事实源。

## 异常与边界

- GET 失败或 D1 未绑定：显示共享数据加载失败，保持写入锁定。
- 服务器没有共享状态：不自动用默认演示数据初始化；需受控初始化。
- 旧客户端不提交 `baseUpdatedAt`：返回 409 `SHARED_STATE_BASE_REQUIRED`。
- 版本落后：返回 409 `SHARED_STATE_VERSION_CONFLICT`，保留本地修改并提示刷新。
- 写入或审计失败：返回失败，不清除本地脏标记；待完成审计标记为失败。
- 多标签页：第一个成功写入的标签页推进版本，其他标签页的旧版本写入失败。

## 验收标准

- 带默认状态和脏标记的旧页面打开后，不产生 D1 写入。
- 未先 GET 的新页面也不能 POST。
- 两个客户端从同一版本读取后，先写者成功，后写者收到 409。
- 同一个标签页仅发生组织缓存时间刷新时不发出 POST，也不新增快照或审计。
- 每次成功写入新增一个写前快照和一条成功审计。
- 生产恢复后显示 3 个产品、26 个任务、4 个交付物、2 个产品计划和 20 条评审记录。
- 恢复审计为 `audit_e24437c8-85fe-4d28-8849-a7ad29088da2`，且恢复后的业务计数保持不变。
- 恢复前后组织成员、商品目录和其他业务表计数不变。

## 上线与回滚

- 先部署服务端版本门禁，使旧客户端立即失去写权限；再写回恢复状态。
- 上线前保留事故后完整 SQL 导出、事故前状态 JSON、Time Travel 书签及 SHA-256。
- 代码回滚可恢复上一 Pages 部署，但不得恢复会取消版本门禁的旧 API，除非同时进入只读维护。
- 数据回滚使用本次恢复审计关联的写前快照，不执行整库 Time Travel 回滚。
