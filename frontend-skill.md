# Frontend Skill：规范化产品型网页设计与实现

## 适用场景

用于设计和实现 SaaS、内部系统、流程工具、数据看板、CRM、ERP、项目管理、商品/产品生命周期系统等产品型网页。

目标不是做漂亮截图，而是做一个清晰、稳定、可扩展、可操作、可验收的前端界面。

## 核心判断

产品型网页的前端质量主要看 6 件事：

1. 信息架构是否清楚
2. 组件是否一致
3. 状态是否明确
4. 操作是否可预期
5. 响应式是否不崩
6. 可访问性是否合格

## 设计前必须先定

### 1. 用户和任务

先写清楚：

- 谁在用
- 每天打开页面要做什么
- 最常见的 3 个操作是什么
- 哪些信息必须第一眼看到
- 哪些操作是高风险操作

内部工具不要从视觉风格开始，先从任务流开始。

### 2. 页面优先级

页面第一屏必须回答：

- 当前对象是什么
- 当前状态是什么
- 有什么风险
- 下一步该做什么

如果第一屏只能看到欢迎语、介绍文案、装饰图，这个页面不合格。

## 推荐信息架构

产品型系统优先使用：

- 左侧一级导航
- 顶部页面标题和主操作
- 指标概览
- 主工作区
- 右侧状态/详情/风险区
- 表格、看板、列表、弹窗、抽屉

避免：

- 大 hero
- 营销式分屏
- 复杂装饰背景
- 卡片套卡片
- 过多解释性文案

## 视觉规范

### 颜色

使用设计 token，不要在各处随手写颜色。

推荐基础：

- 背景：`#f5f5f7`
- 主文字：`#1d1d1f`
- 次文字：`#6e6e73`
- 边框：`rgba(0,0,0,.08)`
- 主操作：`#0071e3` 或 `#007aff`
- 成功：`#34c759`
- 警示：`#ff9f0a`
- 风险：`#ff3b30`

要求：

- 主色只用于主操作、当前状态、关键进度。
- 状态色必须有语义，不要只为了好看。
- 同一页面不要超过 1 个主色 + 3 个语义色。

### 字体

推荐：

```css
font-family: -apple-system, BlinkMacSystemFont, "SF Pro Text", "SF Pro Display", "PingFang SC", "Microsoft YaHei", sans-serif;
```

字号建议：

- 页面标题：26-32px
- 区块标题：16-18px
- 正文：13-14px
- 辅助说明：12-13px
- 表格：13px

要求：

- 不用负字距。
- 不用 viewport 宽度动态缩放字体。
- 表格和工具区不要用 hero 大字。

### 间距和圆角

- 页面外边距：16-28px
- 面板内边距：14-20px
- 组件间距：8/12/16/24px
- 卡片圆角：8px
- 按钮圆角：8px
- 徽标可用胶囊形

## 组件规范

### 按钮

按钮分 3 类：

- Primary：页面主操作，只保留 1 个明显主按钮
- Secondary：普通操作，白底细边框
- Icon button：工具类操作，优先使用图标

要求：

- 按钮必须有 hover、active、focus。
- 危险操作必须用风险色或确认流程。
- 工具按钮优先用图标，不要堆文字按钮。

### 表单

要求：

- label 必须清楚。
- 输入框 focus 必须明显。
- 错误状态必须靠近字段展示。
- checkbox / radio 不要继承普通 input 的宽高。
- 小控件视觉可以小，但点击目标要有足够间距。

### 状态徽标

状态必须可读：

- 待处理：warning
- 进行中：info
- 已完成：success
- 阻塞/风险：danger
- 暂缓：muted 或 danger

不要只用颜色表达状态，必须保留文字。

### 表格

表格必须支持 4 类任务：

- 找到符合条件的记录
- 对比多条记录
- 查看/编辑单条记录
- 对记录执行操作

要求：

- 表头固定语义清晰。
- 行 hover 明显。
- 操作列靠右或固定位置。
- 状态列不要塞长文案。
- 移动端允许横向滚动，不要硬挤。

### 看板

看板适合流程阶段、状态流转、任务分组。

要求：

- 当前阶段必须高亮。
- 已完成、当前、未开始要能区分。
- 卡片内只放必要信息：标题、状态、负责人、下一步。
- 不要把长文档塞进看板卡片。

## 可访问性底线

必须做到：

- 所有交互控件可键盘访问。
- focus 状态清晰可见。
- 文字和背景有足够对比度。
- 交互目标至少满足 24px 目标或周围有足够间距。
- 图标按钮要有可理解的文本或 `aria-label`。
- 状态不能只靠颜色表达。

## 响应式规范

桌面端优先信息密度，移动端优先可读和不重叠。

要求：

- 1100px 以下导航可折叠或变为顶部图标导航。
- 720px 以下主布局变单列。
- 表格在移动端横向滚动。
- 固定格式元素必须有 `minmax(0, 1fr)`、`overflow-x` 或明确宽度约束。
- 文本不能溢出按钮、徽标、卡片。

## 实现规范

### CSS

必须使用：

- `:root` token 管理颜色、阴影、边框、半径
- `box-sizing: border-box`
- `min-width: 0` 防止 grid/flex 撑破
- `overflow-x: auto` 处理宽表和看板
- 清晰的 hover / active / focus

避免：

- 大量 inline style
- 魔法数字散落
- 组件互相嵌套过深
- 用颜色 class 代替语义 class

### JavaScript

产品原型也要有真实状态流转：

- 创建对象后列表要更新
- 审批后状态要变化
- 归档后交付物要变化
- 转入开发后必须生成对应产品记录
- 页面刷新后本地状态不能丢，除非明确重置版本

## 交付验收

完成前必须检查：

1. 首屏是否直接呈现业务状态
2. 主操作是否明确
3. 是否存在无意义演示按钮
4. 是否所有状态都有文字说明
5. 表格、看板、卡片是否不重叠
6. 移动端是否可用
7. 控制台是否无错误
8. 关键流程是否能跑通
